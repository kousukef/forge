# docs/domain/ -- プロジェクト固有ドメイン知識

## 目的

プロジェクト固有のドメイン知識を構造化して配置する場所。ここに配置された知識は以下のワークフローで自動的に活用される:

- `/brainstorm` 実行時にドメインコンテキストとして参照され、提案の品質が向上する
- `/spec` 実行時に domain-analyzer がドメイン制約を抽出し、Error/Boundary/NFR シナリオの精度が向上する
- `/compound` 実行時に Learning Router がドメインに関する学びを適切なファイルにルーティングする

## 配置すべきファイル

| ファイル | 目的 | 記述例 |
|---|---|---|
| `business-rules.md` | ビジネスルール、業務ロジック、ドメイン固有の制約 | 「注文金額が10万円以上の場合は承認フローが必要」「ユーザーの退会後30日間はデータを保持する」 |
| `domain-model.md` | ドメインモデルの主要概念、エンティティ間の関係、状態遷移 | 「注文のステータスは draft -> confirmed -> shipped -> delivered の順に遷移する」 |
| `stakeholders.md` | ステークホルダーの一覧、優先度、各ステークホルダーの要求・制約 | 「営業チーム: ダッシュボードのレスポンスタイムを3秒以内に保つことが最優先」 |
| `tech-constraints.md` | 技術的制約、SLA、パフォーマンス要件、セキュリティ要件 | 「API レスポンスタイムは p99 で 500ms 以内」「PII データは暗号化して保存」 |
| `runbooks/` | 運用手順書、障害対応フロー、デプロイ手順 | 「DB マイグレーション手順」「障害発生時のエスカレーションフロー」 |

## 生成方法

### /forge-init による自動生成

`/forge-init` コマンドを実行すると、ソクラテス式対話を通じてプロジェクトのドメイン知識が収集され、対応するファイルが自動生成される。情報が収集された領域のみファイルが作成される。

### /compound による継続的更新

開発中に蓄積された学びは `/compound` 実行時に Learning Router がドメイン分類テーブルに基づいて自動的に適切なファイルにルーティングする。

## 空ファイル禁止原則

`docs/domain/` 配下に空のファイルやテンプレートのみのファイルを配置してはならない。各ファイルは実際のドメイン知識が収集された場合にのみ作成する。空のファイルを大量に用意しても運用で放棄されるため、内容がある場合にのみファイルを生成する設計としている。
