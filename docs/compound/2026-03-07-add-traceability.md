# トレーサビリティ基盤導入での仕様品質パターン

## 何が起きたか

6ファイル・12要件にまたがる Markdown ドキュメント変更（spec-writer, spec-validator, implementer, compound, spec, CLAUDE.md）を実施。実装フェーズで 2 件の軽微な不整合が self-correct され、最終検証・レビューともに指摘 0 件で完了した。

## なぜ起きたか

仕様段階で以下を徹底したことが品質に寄与:
1. proposal.md の「Open Questions」で未解決問題を事前列挙し、design.md で根拠付き決定
2. design.md に「ID 体系テーブル」「フォーマット定義」「マージ方式」を詳細に記述
3. 既存エージェント定義のパターンを踏襲し、「既存セクション内の新規項目」として追加

## どう解決したか

最終検証で発見された 2 件の不整合:
1. spec-validator.md の Step 3 見出し「9つの検証項目」→「10の検証項目」更新漏れ
2. spec.md の Phase 3 記述の検証項目数更新漏れ

いずれも「同一概念がドキュメント内の複数箇所で言及される」パターンで発生。最終検証タスクで検出・修正。

## 教訓

- 複数ファイル変更では「ファイル間整合性テーブル」を仕様に含めると不整合を防げる
- Phase B（振り返り）の「想定外の発見」が全タスクで「なし」＝仕様品質の良い指標
- Markdown 変更でもコード実装と同等の複雑性を持つ。仕様粒度を過小評価しない
- リスト挿入時の「既存項目の番号繰り下げ」は仕様に明示すべき
