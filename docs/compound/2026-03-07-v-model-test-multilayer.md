# V-Model テスト多層化: 仕様品質と Teams モード並列実装の学び

## 何が起きたか

Phase 2（V-Model テスト多層化）で、commands/test.md の多層ワークフロー化、commands/spec.md と agents/spec/spec-writer.md の受入テスト計画テンプレート追加、agents/spec/spec-validator.md の検証項目追加（10→11）を5ファイル・12要件で実施した。Teams モードで3つの implementer を並列起動し、10タスクを同時処理した。

## なぜ起きたか

### うまくいったこと

1. **Phase 1 の教訓を仕様に反映した結果、実装での曖昧性がほぼゼロに**: ファイル間整合性テーブル、横断チェックタスク（Task 10）、検証項目数の同期対象の明示的列挙により、implementer の解釈判断が不要な仕様が実現できた。レビュー指摘ゼロの直接原因

2. **3層テスト構造の採用判断が正しかった**: proposal の4層（L1-L4）からリサーチ結果に基づき3層（L1/L2/L3）に変更。Forge が扱うプロジェクト規模の多様性を考慮し、テーラリングルールで柔軟性を確保

### 問題が起きたこと

3. **Teams モードでの traceability.md 同時編集競合**: 3つの implementer が並列で traceability.md の Forward/Backward Traceability テーブルを更新した結果、`replace_all` で意図しない行が置換される競合が発生。手動修正で解消したが、チーム並列実装時の共有ファイル管理に課題あり

4. **spec-validator 新規検証項目の検証レベル（ブロッキング/非ブロッキング）が仕様で未指定**: 検証項目 11（受入テスト計画の網羅性チェック）のブロッキング/非ブロッキングが delta-spec に明記されておらず、implementer が独自判断でブロッキング（通常レベル）として実装。仕様書のデフォルト検証レベルが未定義

## どう解決したか

- 問題3: implementer が手動で競合を検出・修正。横断チェックタスク（Task 10）で最終的な整合性を確認
- 問題4: 既存項目10（トレーサビリティ）が「警告レベル」と明記されていることとの対比で、明記なし = ブロッキングと判断。仕様の意図としては妥当な解釈

## 教訓

1. **仕様に前回の教訓を反映すれば、レビュー指摘ゼロは達成可能**: ファイル間整合性テーブル + 横断チェックタスクのパターンを標準化すべき
2. **Teams モード並列実装時、traceability.md のような共有ファイルは最後に一括更新すべき**: 各 implementer が個別に更新すると競合する。横断チェックタスクに一括更新を集約するか、traceability.md 更新を専任タスクに分離する
3. **spec-validator の検証項目追加時は検証レベル（ブロッキング/非ブロッキング）を必ず明記すべき**: デフォルトの検証レベルが定義されていないため、implementer の解釈に依存してしまう
