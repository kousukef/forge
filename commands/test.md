# /test コマンド

## 目的

テストスイートを実行し、結果を検証する。修正が機能することを証明するまで完了としない。

## テストレベル定義

| レベル | 名称 | テスト対象 | V-Model 左辺対応 | フィードバック |
|---|---|---|---|---|
| L1 | Unit | 個別モジュール（ユニットテスト + 静的解析 + ビルド） | 詳細設計 (tasks.md) | 秒〜分 |
| L2 | Integration | モジュール間連携（結合テスト + E2E） | アーキテクチャ設計 (design.md) | 分〜10分 |
| L3 | Acceptance | ユーザーストーリー充足（受入テスト計画の検証） | 要件定義 (proposal.md) | 手動含む |

## Entry/Exit Criteria

| レベル | Entry Criteria | Exit Criteria |
|---|---|---|
| L1 | なし（常に実行可能） | 全ユニットテスト合格、静的解析エラーゼロ、ビルド成功、カバレッジ閾値達成（設定されている場合） |
| L2 | L1 の Exit Criteria 達成 | 全結合テスト/E2E テスト合格 |
| L3 | L2 の Exit Criteria 達成（L2 スキップ時は L1 の Exit Criteria 達成） | 全ユーザーストーリーの受入テストシナリオが検証済み |

## テーラリングルール

- **L1**: 全プロジェクトで必須（スキップ不可）
- **L2**: プロジェクトに結合テストまたは E2E テストが存在する場合に実行。存在しない場合は「L2 スキップ: 結合テスト/E2E テストが定義されていません」と記載してスキップ
- **L3**: design.md に「受入テスト計画」セクションが存在する場合に実行。存在しない場合は「L3 スキップ: 受入テスト計画が design.md に定義されていません」と記載してスキップ

## --level 引数

`/test --level <L1|L2|L3>` でテスト実行の上限レベルを指定できる:

- `--level L1`: L1 のみ実行。L2, L3 はスキップ
- `--level L2`: L1 と L2 を順に実行。L3 はスキップ（L1 は L2 の Entry Criteria のため常に実行）
- `--level L3`: L1, L2, L3 を順に実行（引数なしと同等の動作）
- 無効な値（例: `--level L4`）: 「無効なレベル指定: L4。有効な値は L1, L2, L3 です」とエラーメッセージを出力し、テストは実行しない

## ワークフロー

### Step 1: テーラリング判定

1. **テスト資産検出**: プロジェクトのテストランナー、E2E フレームワーク、受入テスト計画（design.md の「受入テスト計画」セクション）の存在を確認
2. **--level 引数チェック**: 指定がある場合は有効値（L1, L2, L3）か検証。無効値の場合はエラーメッセージを出力して終了
3. **実行レベル決定**: テーラリングルールと --level 引数に基づき、実行するレベルを決定
   - L1: 常に実行
   - L2: テーラリングルールで実行可能 かつ --level が L2 以上（または指定なし）の場合に実行
   - L3: テーラリングルールで実行可能 かつ --level が L3 以上（または指定なし）の場合に実行

### Step 2: L1 実行（Unit）

以下を順番に実行する:

1. **ユニットテスト実行**: プロジェクトのテストランナーを実行
2. **静的解析**: プロジェクトの静的解析ツールを実行（型チェッカー、リンター等）
3. **ビルド検証**: プロジェクトのビルドコマンドを実行
4. **カバレッジ確認**: プロジェクトのカバレッジ閾値（設定されている場合）を達成しているか確認

L1 の Exit Criteria を満たさない場合、L2 以降はスキップし Step 5 へ進む。

### Step 3: L2 実行（Integration）

**Entry Criteria**: L1 PASS

以下を実行する:

1. **結合テスト実行**: プロジェクトの結合テストを実行
2. **E2E テスト実行**: プロジェクトの E2E テストランナーを実行

L2 の Exit Criteria を満たさない場合、L3 はスキップし Step 5 へ進む。

### Step 4: L3 実行（Acceptance）

**Entry Criteria**: L2 PASS（L2 スキップ時は L1 PASS）

1. **受入テスト計画の読み込み**: design.md の「受入テスト計画」セクションから検証シナリオを取得
2. **US 別集計準備**: traceability.md の US → TP マッピングを Source of Truth として使用。traceability.md が存在しない場合はシナリオ単位の検証のみ実施
3. **各シナリオの検証**: GIVEN/WHEN/THEN 形式の各シナリオを検証し、合格/失敗を判定
4. **US 別集計**: traceability.md のマッピングに基づき、US 単位で合格/失敗を集計

### Step 5: 結果分析

- 全レベルがパスした場合: 結果レポートを出力
- 失敗またはスキップがある場合: レベル別に結果を分析し、根本原因を特定

### Step 6: 失敗時の修正

1. 失敗したテストの根本原因を分析
2. 修正を提案
3. 修正を適用
4. **修正の影響範囲に基づき再実行レベルを決定**:
   - 修正がユニットテスト対象の場合: L1 から再実行
   - 修正が結合テスト/E2E テスト対象の場合: L2 から再実行
   - 判定が困難な場合: L1 から再実行（安全側に倒す）
5. 全てパスするまで繰り返し

## verification-before-completion

- テストが全てパスするまで「完了」と宣言しない
- 失敗したテストがある場合、根本原因を分析して修正を提案
- 修正後に再度全テストを実行して回帰がないことを確認
- 「テストがパスしました」ではなく、**テスト実行結果を実際に貼り付けて**証明する

## 結果レポート形式

```markdown
# テスト結果レポート

## サマリー
- L1 (Unit): PASS / FAIL
- L2 (Integration): PASS / FAIL / SKIPPED
- L3 (Acceptance): PASS / FAIL / SKIPPED

## L1: Unit テスト結果
### ユニットテスト
[実行結果]
### 静的解析
[実行結果]
### ビルド
[実行結果]
### カバレッジ
[カバレッジ結果]

## L2: Integration テスト結果
### 結合テスト
[実行結果]
### E2E テスト
[実行結果]

## L3: Acceptance テスト結果
### US 別受入テスト結果

> traceability.md の US → TP マッピングを Source of Truth として集計する。
> traceability.md が存在しない場合は「traceability.md が存在しないため US 別集計を行えません。シナリオ単位の結果のみ出力します」と記載する。

| US | ストーリー | シナリオ数 | 合格数 | 結果 |
|---|---|---|---|---|
| US-001 | ... | N | N | PASS/FAIL |

> US に対応する TP が1件も存在しない場合は結果を「N/A（テスト観点未割当）」と記載し、警告を出力する。

### 受入テストシナリオ詳細
[各シナリオの検証結果]

## 失敗分析（該当する場合）
### [テスト名]
- **レベル**: L1/L2/L3
- **原因**: [根本原因]
- **修正**: [修正内容]
```

## 禁止事項

- テストを実行せずに「テストは通るはずです」と言うこと
- エラーを無視して「完了」と宣言すること
- 一部のテストをスキップすること
- テストを無効化して通過させること
