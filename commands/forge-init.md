# /forge-init コマンド

REQUIRED SKILLS:
- iterative-retrieval
- verification-before-completion

## 目的

既存プロジェクトに Forge を導入する際に、ソクラテス式対話で AI にドメイン知識を移転し、`openspec/project.md` と `docs/domain/` 配下のドメイン知識ドキュメントを生成する。

5フェーズ構成: 自動分析 -> ソース提供 -> コア5質問 -> 深掘り質問（オプション） -> ドキュメント生成。

## K1-K15 知識領域定義

| ID | 領域名 | 説明 |
|---|---|---|
| K1 | ビジネスルール | 業務ロジック、ドメイン固有のルール、バリデーション条件 |
| K2 | ドメインモデル | 主要エンティティ、集約、値オブジェクト、状態遷移 |
| K3 | ステークホルダー | 関係者、意思決定者、優先度 |
| K4 | ユーザーペルソナ | ユーザー種別、利用シナリオ、期待値 |
| K5 | 技術スタック | 言語、フレームワーク、ライブラリ、バージョン |
| K6 | 技術的制約 | レガシー互換性、パフォーマンス制約、インフラ制約 |
| K7 | DB スキーマ | テーブル構造、リレーション、インデックス |
| K8 | API 設計 | エンドポイント、認証方式、レスポンス形式 |
| K9 | テスト戦略 | テストフレームワーク、カバレッジ方針、CI 構成 |
| K10 | 運用知識 | デプロイ手順、モニタリング、障害対応 |
| K11 | セキュリティ要件 | 認証・認可、暗号化、コンプライアンス |
| K12 | パフォーマンス要件 | SLA、レイテンシ要件、スループット |
| K13 | UI/UX ガイドライン | デザインシステム、アクセシビリティ、ブランドガイド |
| K14 | 外部連携 | 外部 API、SaaS 連携、Webhook |
| K15 | 規制・コンプライアンス | 法的要件、業界規制、データ保護 |

## 再実行時の動作

`/forge-init` が既に実行済みで `docs/domain/` や `openspec/project.md` にファイルが存在する場合:

1. Phase 1 で既存の `docs/domain/` と `openspec/project.md` を読み込み、K1-K15 のカバレッジを算出する
2. カバレッジが低い（情報なし）領域を特定する
3. Phase 2-3 は該当領域に絞った質問のみ実施する
4. Phase 4 は通常通り60%未満の領域について深掘りする
5. Phase 5 では既存ファイルへの**差分追記**を提案する（上書きしない）

## ワークフロー

### Phase 1: 自動分析

**進捗表示**: `[Phase 1/5] 自動分析 -- プロジェクト構造を解析しています...`

codebase-analyzer を Sub Agent として起動し、プロジェクトを自動解析する。

**起動プロンプト**:
```
プロジェクトの以下の情報を分析してください:
- 技術スタック（言語、フレームワーク、主要ライブラリとバージョン）(K5)
- 技術的制約（レガシーコード、非推奨API の使用等）(K6)
- DB スキーマ（テーブル構造、リレーション）(K7)
- API 設計（エンドポイント一覧、認証方式）(K8)
- テスト構成（フレームワーク、カバレッジ）(K9)
- 運用構成（デプロイ設定、CI/CD）(K10)
- 外部連携（外部 API、SaaS 連携）(K14)
- 規制関連（ライセンス、コンプライアンス設定）(K15)
```

**完了後**:
- codebase-analyzer の解析結果を受け取る
- K5-K8, K10, K14, K15 のカバレッジを暫定的に記録する
- 解析結果のサマリーをユーザーに表示する

**エラー時**: codebase-analyzer がエラーで結果を返さない場合、自動分析結果なしで Phase 2 に進む。「自動分析をスキップしました。手動での情報提供をお願いします。」と表示する。

`Phase 1 完了。次のフェーズに進みますか？` と確認する。

---

### Phase 2: ソース提供

**進捗表示**: `[Phase 2/5] ソース提供 -- 既存ドキュメントを収集しています...`

ユーザーに既存ドキュメントの提供を求める:

```
プロジェクトに関する既存のドキュメントがあれば提供してください。
例:
- README やプロジェクト概要
- Wiki や仕様書
- 社内ドキュメントやオンボーディング資料
- ER 図やアーキテクチャ図の説明

テキストとして貼り付けるか、ファイルパスを指定してください。
ドキュメントがない場合は「なし」と回答してください。
```

**ドキュメント提供時**:
- 提供されたドキュメントを解析する
- K1-K15 の各領域についてどの情報がカバーされたかを記録する
- Phase 1 の結果と合わせて不足知識領域を特定する

**「なし」と回答された場合**: Phase 2 をスキップし、Phase 3 に進む。

`Phase 2 完了。次のフェーズに進みますか？` と確認する。

---

### Phase 3: コア5質問

**進捗表示**: `[Phase 3/5] コア質問 -- ソクラテス式対話で知識を収集しています...`

以下の5つの質問を**一つずつ**実施する。**一度に複数の質問を投げない。**

可能な限り**選択肢形式**を含め、回答しやすくする。

#### Q1: ビジネスルールの核心（K1 対象）

```
このプロジェクトで最も重要なビジネスルールは何ですか？

例えば:
A) 課金・決済に関するルール（金額計算、税率、割引条件等）
B) ユーザー権限・アクセス制御のルール
C) データのライフサイクル（保持期間、削除条件等）
D) ワークフロー・承認フロー
E) その他（具体的に教えてください）

最も重要なものを1-3個挙げ、それぞれ簡潔に説明してください。
```

#### Q2: ドメインモデルの主要概念（K2 対象）

```
このプロジェクトの中心となるデータモデル（主要な「もの」や「概念」）は何ですか？

Phase 1 の解析で以下のモデルを検出しました:
[Phase 1 の DB スキーマ分析結果があれば挿入。なければこの行を省略]

以下について教えてください:
- 最も重要なエンティティ（ユーザー、注文、商品 等）とその関係
- 状態を持つエンティティの状態遷移（例: 注文 -> 確定 -> 発送 -> 完了）
- 特別な値の制約（例: 金額は0以上、ステータスは3種類のみ）
```

#### Q3: ステークホルダーの優先度（K3, K4 対象）

```
このプロジェクトの主要なステークホルダーとユーザーについて教えてください。

A) プロジェクトオーナー / 意思決定者は誰ですか？
B) 主なユーザー層は？（例: 社内ユーザー、一般消費者、B2B顧客）
C) 最も優先すべきユーザー体験は？

複数のユーザー種別がある場合、優先度順に挙げてください。
```

#### Q4: 既知の技術的負債（K6 対象）

```
現在のプロジェクトで認識している技術的負債や課題はありますか？

例えば:
A) レガシーコードや古いライブラリへの依存
B) テスト不足の領域
C) パフォーマンスのボトルネック
D) セキュリティ上の懸念
E) スケーラビリティの課題
F) 特になし

該当するものを選択し、詳細を教えてください。
```

#### Q5: 運用上の制約（K10 対象）

```
運用上の制約や重要なルールはありますか？

例えば:
A) デプロイの頻度やタイミングの制約
B) ダウンタイムの許容範囲
C) 監視・アラートの要件
D) バックアップ・リカバリの要件
E) 特になし

該当するものを選択し、詳細を教えてください。
```

**回答が非常に短い場合（1行以下）**: その領域を Phase 4 の深掘り質問候補に追加する。

`Phase 3 完了。次のフェーズに進みますか？（Phase 4 はオプションの深掘り質問です。スキップも可能です）` と確認する。

---

### Phase 4: 深掘り質問（オプション）

**進捗表示**: `[Phase 4/5] 深掘り質問 -- カバレッジが不足している領域を補完しています...`

Phase 1-3 の結果から K1-K15 のカバレッジを算出する。

**カバレッジ計算**: `カバー率 = 情報が1項目以上収集された領域数 / 15`

カバレッジが60%未満の場合、不足領域について追加質問を動的に生成する。

**追加質問の生成ルール**:
- 情報が収集されていない K1-K15 の領域を特定する
- 各領域について1-2問の具体的な質問を生成する
- **一度に1つだけ質問する**
- ユーザーが「十分」と回答したら Phase 4 を終了する

**スキップ条件**:
- カバレッジが60%以上の場合: 「カバレッジは十分です（XX%）。Phase 5 に進みます。」と表示し、Phase 5 に進む
- ユーザーが「十分」「スキップ」等と回答した場合: Phase 4 を終了し、Phase 5 に進む

`Phase 4 完了。ドキュメント生成に進みます。` と表示する。

---

### Phase 5: ドキュメント生成

**進捗表示**: `[Phase 5/5] ドキュメント生成 -- 収集した知識からドキュメントを生成しています...`

project-knowledge-writer を Sub Agent として起動する。

**起動プロンプトに含める情報**:
- Phase 1 の自動分析結果（全文）
- Phase 2 のドキュメント解析結果（あれば）
- Phase 3 のコア5質問の回答（全文）
- Phase 4 の深掘り質問の回答（あれば）
- K1-K15 の暫定カバレッジ状態
- 既存の `openspec/project.md` の有無（再実行時は差分追記モードを指示）
- 既存の `docs/domain/` ファイルの一覧（再実行時は差分追記モードを指示）

**完了後**:
- project-knowledge-writer が生成したファイル一覧を受け取る
- K1-K15 カバレッジサマリーを受け取る
- 最終レポートをユーザーに表示する

## /brainstorm との競合回避

- `openspec/project.md` が既に存在する場合: Phase 5 で project-knowledge-writer に「既存ファイルへの差分追記」を指示する（上書きしない）
- `/forge-init` が先に実行されて `openspec/project.md` が生成された場合: `/brainstorm` の OpenSpec 初期化ステップ（`openspec/project.md` 自動生成）はスキップされる（/brainstorm の既存ロジックで対応済み: 「`openspec/project.md` が存在しない場合は自動生成する」）

## 最終レポート形式

Phase 5 完了後に以下を表示する:

```
/forge-init 完了

生成ファイル:
- openspec/project.md [新規作成 or 差分追記]
- docs/domain/business-rules.md (K1)
- docs/domain/domain-model.md (K2)
- ... (収集された領域のみ表示)

K1-K15 カバレッジサマリー:
| ID | 領域 | カバー |
|---|---|---|
| K1 | ビジネスルール | 済 |
| K2 | ドメインモデル | 済 |
| K3 | ステークホルダー | 未 |
| ... | ... | ... |

カバー率: X/15 (XX%)

次のステップ:
- カバレッジが不足している領域は `/forge-init` を再実行して補完できます
- 機能開発を始めるには `/brainstorm <topic>` を実行してください
```

## 重要なルール

- **一度に1つだけ質問する**（複数質問を一度に投げない）
- 可能な限り**選択肢形式**で質問する
- **空ファイル禁止**: 情報が収集された領域のみファイルを生成する
- **上書き禁止**: 再実行時は既存ファイルに差分追記する
- 各フェーズの開始時に進捗表示を行う
- Phase 4 はオプション。ユーザーが即座にスキップ可能
