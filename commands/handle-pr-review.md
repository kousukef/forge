# Handle PR Review Comments

PRに対するレビューコメントを確認し、必要な修正を行い、コミット・プッシュ・返信を行います。

PR Number: #$ARGUMENTS

## Step 1: Get PR Review Comments

まず、PRのレビューコメントを取得します。

```bash
gh pr view $ARGUMENTS --comments --json reviews,reviewThreads
```

## Step 2: Analyze Review Comments

各レビューコメントを確認し、対応が必要なものを特定します。

**判断基準:**

- バグ修正の指摘 → 対応必須
- コードスタイル/品質の指摘 → 基本的に対応
- 提案/オプション的な指摘 → 妥当性を判断して対応
- 質問のみのコメント → 返信のみで対応

## Step 3: Implement Fixes

対応が必要なコメントについて、一つずつ修正を実装します。

**重要なルール:**

- 各修正は関連するファイルと内容に基づいて行う
- 修正の根拠となるレビューコメントを把握しておく

## Step 4: Create Descriptive Commits

**CRITICAL: コミットメッセージのルール**

❌ 禁止: 「レビューコメントの対応」のような曖昧なメッセージ
✅ 必須: 具体的に何を変更したかを記述

**コミットメッセージ例:**

- `fix: null チェックを追加して型安全性を向上`
- `refactor: 冗長な条件分岐を簡潔に修正`
- `style: 変数名をより明確な名前に変更`
- `perf: 不要な再レンダリングを防止するためメモ化を追加`

各レビューコメントへの対応は、内容に応じて個別のコミットにするか、関連するものをまとめてコミットします。

## Step 5: Push All Commits

すべてのコミットが完了したら、プッシュします。

```bash
git push
```

## Step 6: Reply to Review Comment Threads

**CRITICAL: 返信のルール**

❌ 禁止: PRのルートコメントとして追加
✅ 必須: 各レビューコメントのスレッドに直接返信

**返信方法:**

```bash
# スレッドへの返信（conversation ID または review comment ID を使用）
gh api repos/{owner}/{repo}/pulls/{pr}/comments/{comment_id}/replies -f body="返信内容"
```

**返信内容の例:**

- 修正した場合: `修正しました。[該当コミットハッシュ] で対応しています。`
- 対応しない場合: `[理由を説明] のため、今回は対応を見送りました。ご意見ありがとうございます。`
- 質問への回答: 質問に対する適切な説明

## Step 7: 学習ループ

> **注意**: 本ステップは外部PRレビュー（人間/bot）の指摘に対する学習ループです。`/review` の Step 7（Forge LLMレビュー指摘に対する学習ループ）とは異なります。記録先はプロジェクト側であり、Forge 本体は変更しません。

**スコープ制限**: プロジェクト側のファイルのみを対象とする。Forge 本体（`~/.claude/` 配下や Forge リポジトリ自体）は変更しない。

### スキップ判定

`$ARGUMENTS` に `--no-learn` フラグが含まれているか確認する。

- `--no-learn` が含まれている場合: 「学習ループをスキップしました（--no-learn）」と出力し、Step 7 を終了する
- `--no-learn` が含まれていない場合: 以下のサブステップを実行する

### [7a] レビュー指摘の分析・種別分類・学び抽出

進捗メッセージ: 「[7a] レビュー指摘を分析中...」

**指摘ゼロ件の判定**: Step 2 で分析したレビューコメントのうち、修正対応を行ったもの（質問のみのコメントを除く）がゼロ件の場合、「記録可能な学びが抽出されませんでした」と出力し、Step 7 を正常終了する。

**指摘分析**: Step 2 で分析したレビューコメントと Step 3 で実施した修正内容を照合し、各指摘について以下を整理する:

- 指摘の要約
- 修正内容の要約
- 指摘の種別（下記参照）

**種別分類**: 各指摘を以下の7種別のいずれかに分類する:

1. **コーディングパターン** -- 命名規則、コードスタイル、イディオム、型安全性
2. **設計・アーキテクチャ** -- モジュール構成、責務分離、API設計、依存関係
3. **テスト不足** -- テストケースの欠落、カバレッジ不足、テスト品質
4. **ドキュメント不整合** -- コメント・README・仕様との不一致
5. **セキュリティ** -- 脆弱性、認証・認可、入力検証
6. **パフォーマンス** -- 計算量、メモリ使用、不要な処理
7. **その他** -- 上記に該当しない指摘

分類が困難な場合は「その他」に分類し、指摘の原文を学び記録に含める。

**学び抽出**: 各指摘（または類似パターンをグルーピングしたもの）から以下の3観点で学びを抽出する:

- **何が問題だったか**: 指摘された問題の説明
- **なぜ発生したか**: 根本原因の分析
- **どう防止できるか**: 今後の防止策

指摘が10件以上の場合は、類似パターンをグルーピングして学びを抽出する。

### [7b] 学びの記録

進捗メッセージ: 「[7b] 学びを記録中...」

**記録先の動的判断**: プロジェクトの既存ディレクトリ構造を以下の優先順で走査し、最初に見つかったディレクトリを記録先とする:

1. `docs/compound/`
2. `docs/learnings/`
3. `.claude/learnings/`
4. `docs/`

- 候補が複数見つかった場合は、最も適切な候補を提示し、ユーザーに確認する
- いずれも存在しない場合は `docs/compound/` ディレクトリの作成を提案し、ユーザー確認後に作成する。ユーザーが拒否した場合は代替パスを尋ねる。代替パスも指定されない場合は記録をスキップし、「学びの記録をスキップしました（記録先が指定されませんでした）」と出力する

**ファイル名形式**: `YYYY-MM-DD-pr-<pr-number>-learnings.md`

- 同名ファイルが既に存在する場合は、既存ファイルに追記するか新規ファイル（サフィックス `-2`, `-3` 等）を作成するかをユーザーに確認する

**記録フォーマット**: プロジェクトに既存の学び記録形式がある場合はそちらを優先する。ない場合は以下の形式で記録する:

```markdown
---
source: pr-review
pr: <pr-number>
date: YYYY-MM-DD
categories: [種別1, 種別2]
---

# PR #<pr-number> レビューからの学び

## 指摘サマリー

| # | 指摘内容 | 種別 | 修正内容 |
|---|---|---|---|
| 1 | [指摘の要約] | [種別] | [修正の要約] |

## 学び

### [学びのタイトル]
- **何が問題だったか**: [問題の説明]
- **なぜ発生したか**: [根本原因]
- **どう防止できるか**: [防止策]
```

**サマリー出力**: 記録完了後、記録した学びの概要をターミナルに出力する:
- 記録先パス
- 指摘件数と種別の内訳
- 抽出した学びの件数

**エラーハンドリング**: 学びの記録中にファイル書き込みエラーが発生した場合は、エラー内容を出力し、Step 7 を終了する。学びの記録失敗は /handle-pr-review コマンド全体をブロックしない。

### [7c] レビュアー更新提案

進捗メッセージ: 「[7c] レビュアー更新を提案中...」

**スコープ制限**: プロジェクト側の `agents/review/` 配下のファイルのみを対象とする。Forge 本体のレビュアーは変更しない。

**提案の生成**: [7a] で抽出した指摘パターンに基づき、以下の2種類の提案を生成する:

1. **既存レビュアー更新**: プロジェクトの `agents/review/` 配下の既存レビュアーで対応可能な指摘パターンがある場合、該当レビュアーへのチェック項目追加を提案する
2. **新規レビュアー作成**: 既存レビュアーでカバーされていない観点が必要な場合、新規レビュアーの作成を提案する

提案は優先度の高い順に**最大3件**まで表示する。4件以上の候補がある場合は「他 N 件の提案候補があります」と表示する。

**提案の表示形式**:

```
=== レビュアー更新提案 ===

[提案 1] 既存レビュアー更新: <reviewer-name>
  追加チェック項目: <具体的な項目>
  理由: <指摘パターンに基づく根拠>
  -> 承認 (y) / 拒否 (n)?

[提案 2] 新規レビュアー作成: <new-reviewer-name>
  対象: <チェック対象の説明>
  理由: <既存レビュアーでカバーされていない観点>
  -> 承認 (y) / 拒否 (n)?
```

**承認/拒否フロー**: 各提案に対してユーザーが承認/拒否を選択する。承認した提案のみ適用する。

- 既存レビュアー更新が承認された場合: 該当レビュアーファイルにチェック項目を追加する（既存構造を維持）
- 新規レビュアー作成が承認された場合: テンプレートに基づいてファイルを作成する
- ユーザーが全ての提案を拒否した場合: 「レビュアー更新はスキップされました」と出力する

**新規レビュアーテンプレート**: プロジェクトの `agents/review/` 配下に既存レビュアーが存在する場合、そのパターン（frontmatter 構造、セクション構成、出力形式）を踏襲する。存在しない場合は以下の Forge 標準テンプレートを使用する:

```yaml
---
name: <reviewer-name>
description: "<指摘パターンに基づく説明>"
model: opus
tools: [Read, Grep, Glob]
skills: [iterative-retrieval]
---

# <Reviewer Name>

## 役割
[レビュアーの責務]

## 処理手順
[Step N 形式]

## 出力形式
[review-aggregator 入力仕様に準拠: 指摘番号、レビュアー名、優先度、確信度、対象ファイル、指摘内容、推奨修正、関連仕様]
```

**`agents/review/` 不在時のフォールバック**: プロジェクトに `agents/review/` ディレクトリが存在しない場合は、ディレクトリの作成を含めた提案を行い、ユーザー確認を得る。

**エラーハンドリング**: レビュアーファイルの更新中にエラーが発生した場合は、エラー内容を出力し、該当提案をスキップして次の提案に進む。

### [7d] コミット判断

進捗メッセージ: 「[7d] コミット判断...」

**変更なし時のスキップ**: [7b] で学びの記録も [7c] でレビュアー更新も行われなかった場合は、コミット判断の提示をスキップし、Step 7 を正常終了する。

**変更内容の確認**: コミット前に `git diff` で変更内容を表示し、ユーザーが確認できるようにする。

**コミット方法の選択**: ユーザーに以下の選択肢を提示する:

1. **現在のPRに追加コミットとして含める**: 変更をステージングし、コミット・プッシュする
2. **コミットせずワーキングツリーに残す**: 変更をそのまま残し、Step 7 を終了する

**コミットメッセージ形式**: `docs(<scope>): PRレビューからの学びを記録`

- `<scope>` は変更対象に応じて設定する（例: `compound`, `learnings`, `review`）
- 学び記録のみの場合: `docs(<scope>): PRレビューからの学びを記録`
- レビュアー更新も含む場合: `docs(<scope>): PRレビューからの学びを記録しレビュアーを更新`

**git push の実行**: コミット後に `git push` を実行する。

- `git push` が失敗した場合: エラー内容を出力し、「手動で push してください: `git push`」とリカバリ手順を案内する。コミット自体はローカルに残す

## Workflow Summary

1. `gh pr view` でレビューコメントを取得
2. 各コメントを分析し、対応方針を決定
3. 必要な修正を実装
4. **具体的な説明を含むコミットメッセージ**でコミット
5. `git push` でプッシュ
6. 各レビュースレッドに**直接返信**（ルートコメントではなく）
7. レビュー指摘から学びを抽出・記録し、レビュアー更新を提案（`--no-learn` でスキップ）

## Notes

- レビューコメントが `gh api` で取得できない場合は `gh pr view --json` を使用
- pending状態のreviewがある場合は、そのreview threadのcomment IDを正確に特定する
- 返信時は必ずスレッドのコンテキストを維持する
- Step 7 の学習ループはプロジェクト側のファイルのみを対象とする（Forge 本体は変更しない）
- `--no-learn` フラグを指定すると Step 7 の学習ループをスキップできる
- Step 7 内のエラーは /handle-pr-review 全体をブロックしない
