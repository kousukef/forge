# ドリフト補正

目の前の出力を直し、次にシステムそのものを直して再発を防ぐ。

ユーザーが Claude の出力を修正しても、その修正は通常セッションとともに消える。永続的な設定が変わらないため、次の会話で同じズレが再発する。このスキルはそのループを閉じる: ズレの原因を分析し、修正を記録すべき適切な場所を特定し、アーティファクトを作成・更新することで、以降の全セッションに学びを還元する。

## 発動条件

- ユーザーが Claude の出力を修正した（「Xじゃなくて Y にして」）
- ユーザーが持続すべきプリファレンスを述べた（「DBカラムは常に snake_case にして」）
- ユーザーが繰り返し発生する問題に気づいた（「またこれか」）
- ユーザーがルール・スキル・設定の変更を明示的に依頼した
- ユーザーがセッション間で記憶してほしいことを述べた

## コアワークフロー

```
CAPTURE --> ANALYZE --> ROUTE --> DRAFT --> APPLY
 (何が)     (なぜ)    (どこに)  (どう)   (承認)
```

### Step 1: CAPTURE -- ズレを把握する

会話の文脈から直接抽出する。ユーザーに既に述べたことを繰り返させない。不明確な場合のみ質問する。

把握すべき要素:
1. **期待**: ユーザーが望んでいたこと
2. **実際**: Claude が生成したもの
3. **差分**: 具体的に何が違うか

軽微で自明なズレ（1行プリファレンス等）はサマリーを省略し、直接 ROUTE に進んでよい。それ以外の場合、簡潔なズレサマリーを出力する:

```
## ズレの要約
- 期待: [ユーザーの意図]
- 実際: [生成されたもの]
- 差分: [具体的な違い]
```

### Step 2: ANALYZE -- 根本原因

ズレが発生した理由を特定する。根本原因が修正方法を決定する。

| 根本原因 | 説明 | シグナル |
|---|---|---|
| **ガイダンス欠如** | このケースに対するルール・スキル・設定が存在しない | このトピックが初めて出てきた |
| **ガイダンス曖昧** | ルールは存在するが複数の解釈が可能 | 「適切に」「必要に応じて」等の曖昧な表現 |
| **スキル未適用** | 関連スキルは存在するがロードされなかった | スキル名がタスクの文脈と合致していない |
| **ルール競合** | 複数のルールが異なる方向に引っ張っている | 適用可能な2つのルールが矛盾している |
| **暗黙の前提** | ユーザーが明示していないプリファレンスがある | 「当然こうするでしょう」 |
| **コンテキスト喪失** | ガイダンスは存在するがコンテキストウィンドウ内にない | 長い会話、読み込まれなかったファイル |
| **スコープ不一致** | ルールがプロジェクト固有なのにグローバルにしかない、またはその逆 | あるプロジェクトでは動くが別のでは動かない |

自問する: 「適切な指示が適切なタイミングでロードされていたら、このズレは防げたか？」 Yesならアーティファクトの更新で解決できる。Noならモデルの限界かもしれない -- その場合は「このパターンは現在のモデルでは一貫した対応が難しい」と正直に伝え、可能であればフックによる自動チェックなど代替手段を提案する。

### Step 3: ROUTE -- 適切なアーティファクトを選択する

根本原因に基づき、アーティファクトの種別とスコープを選択する。詳細な決定木と例は `references/artifact-routing.md` を参照。

**クイックルーティングガイド:**

| 記録する内容 | アーティファクト | スコープ |
|---|---|---|
| シンプルなプリファレンス（「常にXを使う」） | CLAUDE.md | プロジェクト or グローバル |
| コーディング規約やパターン | `rules/` or `reference/` | プロジェクト |
| 複雑なマルチステップワークフロー | スキル (`.claude/skills/`) | グローバル or プロジェクト |
| 自動チェック・防止機構 | フック (`.claude/hooks/`) | グローバル or プロジェクト |
| エージェントの動作変更 | エージェント定義 (`agents/`) | プロジェクト |
| ドメイン知識・ベストプラクティス | `reference/` ファイル | プロジェクト |
| セッション横断の一時的な知識 | `memory/` (auto memory) | プロジェクト |

**スコープの判断:**
- **グローバル** (`~/.claude/`): 全プロジェクトに適用（個人のプリファレンス、普遍的パターン）
- **プロジェクト** (リポジトリローカル): このコードベースのみに適用（プロジェクト固有の規約、技術スタックのルール）

迷ったらプロジェクトスコープを優先する -- より安全で具体的。

### Step 4: DRAFT -- 修正を生成する

ルーティングの判断に基づきアーティファクトを生成する。種別によりアプローチが異なる:

#### 新規アーティファクト
最小限の焦点を絞った内容でファイルを作成する。この特定のズレを防ぐために必要なことだけを含める。過剰な設計や投機的なルール追加はしない。

#### 既存アーティファクトの更新
1. まず現在のファイルを読む
2. 新しいガイダンスが収まる場所を特定する
3. 必要最小限の変更を追加する
4. 既存の内容と矛盾しないことを確認する

**重要な原則:**
- **理由を説明する**: 「常にXを使え」ではなく「Xを使う。理由は[理由]。これにより[問題]を防げる」と書く
- **具体的にする**: 「適切な命名を使う」ではなく「DBカラムは snake_case、TypeScript 変数は camelCase」と書く
- **例を含める**: 具体的な before/after の例1つは、何段落もの説明に勝る
- **ネガティブ例が助けになる**: 境界が微妙な場合、やってはいけないことを示す

**ドラフトのフォーマット:**

```markdown
## 変更提案

**対象**: [ファイルパス]
**操作**: [新規作成 / 追記 / 修正]
**根本原因**: [Step 2 より]

### 内容

[追加・修正する実際の内容を全文表示]

### 根拠

[この修正がズレを防ぐ理由、1-2文]
```

### Step 5: APPLY -- ユーザー承認ゲート

**アーティファクトの作成・更新は必ずユーザーの明示的な承認を得てから実行する。** 自動適用は禁止。

1. ドラフトを根拠とともにユーザーに提示する
2. **明示的な承認を待つ** -- `AskUserQuestion` で承認を求める。以下を明示:
   - 変更対象ファイルのパス
   - 変更内容（新規作成 / 追記 / 修正）
   - 変更の根拠（根本原因からの論理的つながり）
3. ユーザーの応答に基づいて分岐:
   - **承認**: 変更を適用し、完了サマリーを出力
   - **修正要求**: ドラフトを修正して再度承認を求める
   - **却下**: 変更を破棄。理由を記録し、今後の参考にする
   - **保留**: 変更内容を記録のみして適用しない（ユーザーが後で手動適用できるよう内容を提示）
4. 適用後、変更内容と対象ファイルを簡潔にサマリー出力

承認なしに Write/Edit でアーティファクトを変更してはならない。これはユーザーの設定を勝手に変更しないための安全装置であり、信頼関係の基盤である。

## 複数のズレを扱う場合

1つの修正が複数の関連する問題を明らかにした場合、バッチで対処する:

1. 特定した全てのズレをリストアップする
2. 対象アーティファクトごとにグループ化する（ファイル変更を最小化）
3. 全ての変更をまとめてドラフトする
4. 一貫した更新として適用する

## 例

### 例 1: コーディングスタイルのプリファレンス

```
ユーザー: 「早期リターンにしてって言ったのに、ネストした if-else になってる」

CAPTURE:
  期待: 早期リターンパターン（ガード節）
  実際: ネストした if-else ブロック
  差分: 制御フローのスタイル

ANALYZE:
  根本原因: ガイダンス欠如 -- コーディングスタイルのルールが存在しない

ROUTE:
  アーティファクト: reference/coding-standards.md（プロジェクトスコープ）
  理由: コーディング規約、このコードベースに適用

DRAFT:
  reference/coding-standards.md に追記:
  "## 制御フロー
   ネストした if-else の代わりに早期リターン（ガード節）を使う。
   インデントの深さを減らし、可読性を向上させるため。

   // Good
   function process(input) {
     if (!input) return null;
     if (!input.valid) return { error: 'invalid' };
     return transform(input);
   }

   // Bad
   function process(input) {
     if (input) {
       if (input.valid) {
         return transform(input);
       } else {
         return { error: 'invalid' };
       }
     } else {
       return null;
     }
   }"
```

### 例 2: 暗黙の前提

```
ユーザー: 「全関数にコメント付けるのやめて。非自明なロジックだけにして」

CAPTURE:
  期待: 複雑・非自明なロジックにのみコメント
  実際: 全関数に JSDoc コメント
  差分: 過剰なコメント

ANALYZE:
  根本原因: 暗黙の前提 -- ユーザーは最小限のコメントを好むが、
  Claude はデフォルトで包括的なドキュメントを生成する

ROUTE:
  アーティファクト: CLAUDE.md（プロジェクトスコープ）
  理由: シンプルなプリファレンス、1行ルール

DRAFT:
  プロジェクトの CLAUDE.md に追記:
  "- ロジックが非自明な場合を除き、関数にコメントや JSDoc を追加しない。
    自明なコードに解説は不要。"
```

### 例 3: 繰り返し発生するパターン（スキル更新）

```
ユーザー: 「このプロジェクトは GraphQL なのに REST エンドポイント生成するの何回目？」

CAPTURE:
  期待: GraphQL リゾルバとスキーマ定義
  実際: REST ルートハンドラ
  差分: API パラダイムの不一致

ANALYZE:
  根本原因: スキル未適用または欠如 -- GraphQL スキルがロードされておらず、
  プロジェクトの規約も文書化されていない

ROUTE:
  アーティファクト: CLAUDE.md + 場合によっては新規スキル
  理由: 基本的なプロジェクト規約 + 複雑なワークフロー

DRAFT:
  1. CLAUDE.md に追加: 「このプロジェクトは GraphQL（Apollo Server）を使用。
     REST エンドポイントは絶対に生成しない。全ての API 変更は
     スキーマファースト設計で: まず schema.graphql を更新、次にリゾルバ。」
  2. パターンが十分に複雑であれば、この技術スタック向けの
     GraphQL パターンスキルの作成を提案する。
```

## アンチパターン

- **過剰に規定しない**: 1つのズレに10個のルールは不要。具体的な問題だけを修正する。
- **重複させない**: 新しいアーティファクトを作る前に、関連するルールが既に存在しないか確認する。小さな編集で済むかもしれない。
- **一度きりのケースにルールを作らない**: ユーザーの修正がこの1回のタスクに固有で再発しそうにないなら、出力だけ直す。全てに永続的なルールが必要なわけではない。
- **理由なく MUST/NEVER を追加しない**: 制約が存在する理由を説明する。モデルは命令形の裸の指示より、理由付きのガイダンスをよりよく守る。
- **スコープを忘れない**: あるプロジェクトの CLAUDE.md に属するルールをグローバル設定に入れるべきではない。

## 付録: /compound との関係

このスキルは `/compound` の事後学習システムを補完する。

| 観点 | drift-correction | /compound |
|---|---|---|
| タイミング | リアルタイム、セッション中 | 開発サイクル完了後 |
| トリガー | ユーザーの修正フィードバック | 明示的コマンド |
| スコープ | 単一のズレ修正 | セッション全体の振り返り |
| 学習ソース | ユーザーフィードバック | 設計書、レビュー、解釈ログ |

Forge ワークフロー内で drift-correction を使用すると、その修正は `/compound` の学習抽出でもキャプチャされ、二重フィードバックループが形成される。
