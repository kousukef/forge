## 決定フローチャート

以下の順序で適切なアーティファクトを判定する:

```
1. シンプルな1行のプリファレンスか？
   YES --> CLAUDE.md
   NO  --> 続行

2. コーディング規約やパターンか？
   YES --> ファイル横断的に広く適用されるか？
           YES --> rules/ or reference/
           NO  --> CLAUDE.md（限定的な注記）
   NO  --> 続行

3. 複雑なマルチステップワークフローか？
   YES --> スキル (.claude/skills/)
   NO  --> 続行

4. 自動的にチェック・強制すべきか？
   YES --> フック (.claude/hooks/)
   NO  --> 続行

5. 特定のエージェントの動作を変えるか？
   YES --> エージェント定義 (agents/ or ~/.claude/agents/)
   NO  --> 続行

6. ドメイン知識やベストプラクティスか？
   YES --> reference/ ファイル
   NO  --> 続行

7. セッション横断で覚えておくべき一時的な知識か？
   YES --> memory/ (auto memory)
   NO  --> CLAUDE.md（汎用的な受け皿）
```

## アーティファクト種別の詳細

### CLAUDE.md

**最適な用途**: シンプルなプリファレンス、プロジェクトレベルの規約、手短な1行ルール。

**特性**:
- 常にコンテキストにロードされる（高い視認性、限られたスペース）
- 2つのレベル: プロジェクト (`<repo>/CLAUDE.md`) とグローバル (`~/.claude/CLAUDE.md`)
- 簡潔に保つ -- 全ての行が全会話でコンテキストトークンを消費する

**使用する場面**:
- 「常にXを使う」/「Yは絶対しない」スタイルのプリファレンス
- プロジェクトの技術選択（「このプロジェクトは Bun を使う、npm ではなく」）
- ファイル命名規則
- デフォルト動作のオーバーライド

**フォーマットのガイダンス**:
- 個別ルールには箇条書きを使用
- 関連ルールは見出しの下にグループ化
- 各ルールは1-2行に収める

### rules/

**最適な用途**: コーディング規約、説明と例が必要な規約。

**特性**:
- ルール設定に基づいてロードされる（常時ロード or オンデマンド）
- CLAUDE.md のエントリより長くできる
- 例や根拠を必要とするルールに適している

**使用する場面**:
- before/after の例が必要なコーディングスタイルルール
- エラーハンドリングパターン
- 複数ケースのある命名規則
- セキュリティ要件

### reference/

**最適な用途**: ドメイン知識、ベストプラクティス、オンデマンドで参照されるパターン。

**特性**:
- 常にはロードされない -- 関連する時に読み込まれる
- 長くできる（ガイド、パターン、チェックリスト）
- トピック別に整理（例: `reference/testing.md`, `reference/coding-standards.md`）

**使用する場面**:
- 技術固有のパターン（例: Next.js の規約、Prisma のパターン）
- テスト戦略とパターン
- アーキテクチャガイドライン
- 詳細なスタイルガイド

### スキル (.claude/skills/)

**最適な用途**: 複雑で反復可能なワークフロー、ドメインの専門知識、マルチステッププロセス。

**特性**:
- description のマッチングでトリガーされる（frontmatter の name + description）
- 参照ファイル、スクリプト、アセットを含められる
- 2つのレベル: プロジェクト (`.claude/skills/`) とグローバル (`~/.claude/skills/`)
- トリガー時に SKILL.md 本文がロードされる

**使用する場面**:
- 修正がマルチステッププロセスに関わる
- パターンが構造化されたガイダンスを必要とするほど複雑
- ワークフローが複数プロジェクトに適用される（グローバルスキル）
- 既存スキルがこのケースをカバーしていない（スキルを更新）

**新規スキル作成 vs 既存スキル更新**:
- ズレが既存スキルのカバーするトピックに関連する場合、そのスキルを更新
- 根本的に新しいワークフローの場合、新規スキルを作成
- 既存スキルの小さなギャップの場合、セクションや例を追加

### フック (.claude/hooks/)

**最適な用途**: 自動防止機構、関連するアクションのたびに実行されるべきチェック。

**特性**:
- ツールイベントで自動的に実行される（ツール使用前/後）
- アクションをブロックまたは警告できる
- プログラマティックに簡単にチェックできるルールに最適

**使用する場面**:
- 特定のファイル操作の防止（例: ルートにファイルを作成しない）
- 書き込み中のコードのアンチパターン検出（例: console.log の検出）
- ワークフロールールの強制（例: サーバーには tmux を要求）
- 危険な操作のブロック（例: force push の防止）

### エージェント定義 (agents/)

**最適な用途**: 特定のエージェントロールの動作変更。

**特性**:
- エージェントは特定の役割とツールアクセスを持つ
- ここでの変更はそのエージェントに委任される全タスクに影響する

**使用する場面**:
- エージェントが特定ドメインで一貫してズレを生成する
- エージェントに追加のチェックリスト項目や制約が必要
- エージェントのワークフローに調整が必要

## スコープの決定

### グローバルスコープ (`~/.claude/`)

プリファレンスが **プロジェクトに関係なく** 適用される場合に使用:

- 個人のコーディングスタイルのプリファレンス
- エディタ・ワークフローのプリファレンス
- コミュニケーションスタイルのプリファレンス
- 普遍的なパターン（例: 「常に早期リターンを使う」）

### プロジェクトスコープ (リポジトリローカル)

プリファレンスが **このコードベースに固有** の場合に使用:

- 技術選択（フレームワーク、言語、ライブラリ）
- プロジェクトのアーキテクチャ規約
- チームのコーディング規約
- ドメイン固有のパターン

### 判断のヒューリスティック

自問する: 「全く異なる技術スタックの新しいプロジェクトでも、このルールが欲しいか？」
- YES --> グローバル
- NO  --> プロジェクト
- 場合による --> プロジェクト（安全なデフォルト）

## 既存アーティファクトの探索

新しいアーティファクトを作成する前に、更新すべき既存のものがないか確認する:

1. **トピックで検索**: rules/, reference/, .claude/skills/, CLAUDE.md をズレに関連するキーワードで Grep
2. **CLAUDE.md を確認**: プロジェクトレベルとグローバルレベルの両方
3. **スキルを確認**: スキル名と description の関連性を確認
4. **rules/ を確認**: 関連する規約をスキャン

新規作成より既存アーティファクトの更新を優先する。断片化と矛盾を防ぐため。

## 更新 vs 新規作成

### 更新する場合:
- 既存ファイルが同じトピック領域をカバーしている
- 修正が既存ルールのエッジケースや例外である
- 既存の内容が不完全だが間違ってはいない

### 新規作成する場合:
- 関連する既存ファイルがない
- トピックが独自のファイルを正当化するほど独立している
- 既存ファイルに追加すると焦点がぼやける

### 置換する場合:
- 既存のルールが間違っている（不完全なだけでなく）
- ユーザーが以前のルールを明示的にオーバーライドした

## アーティファクト種別ごとの例

### CLAUDE.md への追加
```markdown
# ズレ: Claude が npm を使う（bun の代わりに）
# 修正: プロジェクト CLAUDE.md に追加

- このプロジェクトはパッケージマネージャーとランタイムに Bun を使用する。全てのコマンドで `npm`/`npx` の代わりに `bun` を使うこと。
```

### rules/ の更新
```markdown
# ズレ: Claude がクラスを生成する（関数の代わりに）
# 修正: rules/core-essentials.md に追記

## コンポーネントスタイル
フックを使った関数コンポーネントを使用する。クラスコンポーネントは使用しない。
React のクラスコンポーネントはレガシーであり、全ての新しいコンポーネントは
ステートとライフサイクルにフックを使った関数コンポーネントでなければならない。
```

### reference/ の新規作成
```markdown
# ズレ: Claude がプロジェクトのエラーハンドリングパターンに従わない
# 修正: reference/error-handling.md を作成

# エラーハンドリングパターン

## API ルート
全ての API ルートハンドラは `withErrorHandler` ラッパーを使用すること。
ルートハンドラで生の try/catch を使用しない。

## Server Actions
throw する代わりに `{ error: ... }` オブジェクトを返す。
呼び出し元が表示を処理する。

[... 例を含む詳細なパターン ...]
```

### スキルの更新
```markdown
# ズレ: テストスキルが結合テストをカバーしていない
# 修正: 既存テストスキルの SKILL.md にセクションを追加

## 結合テスト

テストファイルが複数モジュールからインポートするか
データベースに触れる場合、結合テストパターンを使用する:

[... 詳細なワークフロー ...]
```

### フックの作成
```markdown
# ズレ: Claude がシークレットを含む .env ファイルを作り続ける
# 修正: 書き込み前フックを作成

# フック: block-env-secrets
イベント: PreToolUse (Write)
チェック: 対象ファイルが .env* にマッチする場合、警告してブロック
```

### エージェント定義の更新
```markdown
# ズレ: implementer エージェントがコミット前に型チェックしない
# 修正: implementer エージェント定義に追加

## 完了前チェックリスト
タスク完了を報告する前に:
- [ ] 型チェッカーを実行 (`tsc --noEmit` または同等のもの)
- [ ] 新しい型エラーが導入されていないことを確認
```
