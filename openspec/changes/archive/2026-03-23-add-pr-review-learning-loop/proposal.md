# add-pr-review-learning-loop 提案書

## 意図（Intent）

現在の `/handle-pr-review` はレビュー指摘の修正・コミット・返信までを行うが、指摘から得た学びを記録・活用する仕組みがない。同種の指摘が繰り返し発生しても、防止策が蓄積されない。

本変更は `/handle-pr-review` に学習ループ（Step 7）を追加し、レビュー指摘から学びを自動抽出・記録し、レビュアー定義の更新を提案することで、指摘の再発防止サイクルを組み込む。

## スコープ（Scope）

### ユーザーストーリー

- ユーザーとして、PRレビュー指摘の修正後に学びが自動的に抽出・記録されてほしい。なぜなら同種の指摘の再発を防止したいから。
  - 検証観点: コマンド実行後に記録内容のサマリーが出力される
- ユーザーとして、レビュー指摘に基づくレビュアー定義の更新（既存更新 or 新規作成）がプロジェクト側に提案されてほしい。なぜなら将来のレビューで同じ問題を自動検出したいから。
  - 検証観点: 提案内容がターミナルに表示され、承認/拒否を選択できる
- ユーザーとして、`--no-learn` フラグで学習ループをスキップできるようにしたい。なぜなら軽微な修正時にはオーバーヘッドを避けたいから。
  - 検証観点: フラグ指定時に Step 7 がスキップされ、スキップした旨のメッセージが表示される

### 対象領域

- `commands/handle-pr-review.md`（Step 7 の追加、`--no-learn` 引数の追加）

### 設計上の決定事項

- **記録先**: Forge 本体（`docs/compound/`, `agents/review/`）は変更しない。学びの記録・レビュアー定義はすべてプロジェクト側に記録する
- **記録先の判断**: プロジェクトの既存構造に合わせてコマンドが動的に判断する（固定パスを強制しない）
- **閾値判定なし**: 初回/2回目等の閾値ルールは適用せず、常に学びの記録とレビュアー更新提案を行う（シンプルさ優先）
- **コミット判断**: 学び・レビュアー更新のコミット方法（同一PR/別PR等）はユーザーに都度確認する
- **レビュアー更新**: 既存レビュアーの更新だけでなく、新規レビュアーの作成も提案可能とする
- **デフォルト挙動**: デフォルトで学習ループを実行し、`--no-learn` でスキップ

## スコープ外（Out of Scope）

- Learning Router の閾値ルール（初回/2回/3回の段階的対応）: 常に記録・提案で十分 -- YAGNI
- bot と人間の指摘の区別: どちらも有用なフィードバックとして同等に扱う -- YAGNI
- Forge 本体への学び記録: プロジェクト側のみに記録する設計方針
