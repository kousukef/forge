# add-doc-sync-check 提案書

## 意図（Intent）

実装完了後にプロジェクトのドキュメント更新が漏れるケースが繰り返し発生している。V-Model の左辺（構築: `/implement`）と右辺（検証: `/review`）の両方でドキュメント同期を担保する仕組みを導入し、実装とドキュメントの整合性を自動的に維持する。

関連 Issue: #26, #27

## スコープ（Scope）

### ユーザーストーリー

- 開発者として、`/implement` 完了時にドキュメント更新漏れを自動検出・更新してほしい。なぜならドキュメント同期忘れが繰り返し発生しているから。
  - 検証観点: CLAUDE.md にルールがあるプロジェクトで実装変更後にドキュメントが自動更新される
- 開発者として、`/review` 時にドキュメント同期の過不足と品質をレビューしてほしい。なぜなら実装変更とドキュメントの整合性を検証側でも担保したいから。
  - 検証観点: 重要な実装内容が反映されているかコードと突き合わせて確認する
- 開発者として、`/setup` 時にドキュメント同期ルールを対話的に設定したい。なぜならプロジェクトごとのドキュメント構成に合わせたルールを簡単に作りたいから。
  - 検証観点: `/setup` 実行時にドキュメントディレクトリが検出され、マッピング設定の対話が行われ、CLAUDE.md にルールが追記される

### 対象領域

- `/implement` コマンド定義（Step 5 検証ステップ）
- `/review` コマンド定義（L0 チェック + doc-sync-reviewer エージェント）
- `/setup` コマンド定義（ドキュメント同期ルール設定ステップ）
- `doc-sync-reviewer` エージェント定義（新規）

## 設計判断

- **ルール読み取り元**: CLAUDE.md のみ
- **ルール記載形式**: 自然言語（LLM が文脈で解釈）
- **ルールなし時の挙動**: 毎回軽くスキップログ出力（例: `doc-sync: skipped (no rules found)`）
- **`/implement` 側の更新方式**: implementer が自動更新しコミットに含める
- **`/review` 側の構成**: L0 チェック（更新漏れの有無検出）+ `doc-sync-reviewer`（内容の整合性・品質チェック）
- **`doc-sync-reviewer` の検証観点**: 更新有無 + 実装変更との整合性 + ドキュメント品質（記載漏れ、古い情報の残存等）
- **`/setup` の対話レベル**: ディレクトリ自動検出 + ファイル変更とドキュメントのマッピング設定まで対話で行う

## スコープ外（Out of Scope）

- ドキュメント同期ルールの構造化形式（YAML 等）の規定: 現時点では自然言語で十分 -- YAGNI
- AGENTS.md からのルール読み取り: CLAUDE.md のみで十分 -- YAGNI
- ドキュメント生成（ゼロからの新規作成）: 既存ドキュメントの更新のみが対象
- ドキュメントのフォーマット・スタイル統一: 内容の整合性チェックに留める

## 未解決の疑問点

- `/implement` で自動更新したドキュメントが `/review` の doc-sync-reviewer で不十分と判定された場合のフロー（再更新を implementer に戻すか、レビュー指摘として報告のみか）
