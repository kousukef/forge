# V-Model テスト多層化 提案書

## 意図（Intent）

Forge の `/test` コマンドは現在フラットなテスト実行構造（ユニットテスト → 静的解析 → ビルド → E2E）を持つが、V-Model に基づく多層テスト戦略が欠落している。`/spec` 段階で受入テスト計画を策定し、`/test` を V-Model に基づく多層構造（L1〜L4）に進化させることで、テスト漏れの防止と仕様適合性の検証を強化する。

Phase 1（トレーサビリティ基盤）で導入した US → DD → T → TP のマッピングと連携し、受入テスト結果をユーザーストーリー単位で集計可能にする。

## スコープ（Scope）

### ユーザーストーリー

- US-001: Forge ユーザーとして、`/spec` 実行後に受入テスト計画が design.md に含まれていてほしい。なぜならテストが「後付け」になると検証の質が下がるから。
- US-002: Forge ユーザーとして、`/test` が L1（単体）→ L2（結合）→ L3（システム）→ L4（受入）の多層構造で実行されてほしい。なぜならフラットなテストでは検出できない結合レベルの不具合があるから。
- US-003: Forge ユーザーとして、各テストレベルの実行基準と合格基準が定義されていてほしい。なぜなら基準がなければテストの十分性を判断できないから。
- US-004: Forge ユーザーとして、受入テスト結果がユーザーストーリー単位で集計されてほしい。なぜなら仕様充足の確認をストーリー単位で行いたいから。

### 対象領域

- `commands/spec.md` -- design.md への「受入テスト計画」セクション追加
- `commands/test.md` -- 多層テスト実行ワークフローへの進化
- `agents/spec/spec-writer.md` -- 受入テスト計画の生成ロジック追加
- `agents/spec/spec-validator.md` -- 受入テスト計画の検証項目追加
- `commands/compound.md` -- テスト結果サマリーの学習ソース追加（該当する場合）

## スコープ外（Out of Scope）

- テスト自動生成ツールの導入: テスト戦略の定義が目的であり、自動生成は別テーマ -- YAGNI
- CI/CD パイプラインへの多層テスト統合: Forge は開発フェーズに集中 -- YAGNI
- テストカバレッジの自動計測・レポーティング: 既存の `/test` のカバレッジ確認で十分 -- YAGNI
- L2〜L4 の具体的なテストコード生成: テスト方針の定義が目的であり、実際のテストコード生成は `/implement` の TDD フローで対応 -- YAGNI

## 未解決の疑問点

- L2（結合テスト）とL3（システムテスト）の境界: Forge が扱うプロジェクトの規模は様々であり、全プロジェクトで L2/L3 を明確に区別できるか
- L4（受入テスト）の実行方法: ユーザーストーリーベースの検証は手動確認を伴う場合があり、自動化の範囲をどこまでにするか
- テストレベルのスキップ可否: 小規模な変更で L1 のみで十分な場合に L2〜L4 をスキップできるか
