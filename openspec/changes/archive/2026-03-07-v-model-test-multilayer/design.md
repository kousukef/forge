# V-Model テスト多層化 技術設計

## 概要

Forge の `/test` コマンドを V-Model に基づく3層テスト構造（L1 Unit / L2 Integration / L3 Acceptance）に進化させ、`/spec` で生成する design.md に受入テスト計画セクションを追加する。Phase 1 のトレーサビリティ基盤と連携し、受入テスト結果をユーザーストーリー単位で集計可能にする。

## リサーチサマリー

### 公式ドキュメントからの知見

- **ISTQB v4.0**: テストレベルを4層（コンポーネント/統合/システム/受入）で定義。各レベルに独立した目的・Entry/Exit Criteria を定義すべき
- **Testing Trophy (Kent C. Dodds)**: Integration テストを最も厚くし、Unit は薄く、E2E は最小限が最もコスト効率が良い
- **Microsoft Engineering Playbook**: テストレベルごとに異なるフィードバック速度を設計（L1:秒, L2:分, L3:10分以内, L4:手動含む）

### Web検索からの知見

- **Google Testing Blog**: テストレベルの境界は「依存関係の境界」で定義すべき
- **Spotify Engineering**: 小規模チームでは2層（Fast Tests / Slow Tests）で十分
- **LLM ワークフロー特有の知見**: 構造的プロパティ（ファイル存在、フォーマット準拠）をテストし、内容の完全一致は避ける。Forge のようなワークフローツールでは、テスト対象がコード生成物であるため、この原則が特に重要
- **推奨: 4層ではなく3層が最適**。L2（結合）とL3（システム）の区別は小規模プロジェクトで実質不要。ただしプロジェクトの技術スタックに応じてテーラリング可能にする

## 技術的アプローチ

### 1. テストレベル3層構造の定義

proposal.md の US-002 は L1〜L4 の4層を要望しているが、リサーチ結果に基づき3層を採用する。

| レベル | 名称 | テスト対象 | V-Model 左辺対応 | フィードバック |
|---|---|---|---|---|
| L1 | Unit | 個別モジュール（ユニットテスト + 静的解析 + ビルド） | 詳細設計 (tasks.md) | 秒〜分 |
| L2 | Integration | モジュール間連携（結合テスト + E2E） | アーキテクチャ設計 (design.md) | 分〜10分 |
| L3 | Acceptance | ユーザーストーリー充足（受入テスト計画の検証） | 要件定義 (proposal.md) | 手動含む |

**判断理由**: proposal.md の L2（結合）と L3（システム）は、Forge が扱うプロジェクト規模では実質的に同じレベルのテストを意味する。ISTQB の4層をそのまま適用すると、小規模プロジェクトで L2/L3 の区別に無駄なコストがかかる。3層にすることで Testing Trophy の推奨にも整合する。テーラリングルールにより、プロジェクト規模に応じて L2/L3 をスキップ可能とする。

### 2. /test コマンドの多層ワークフロー

現行の Step 1〜3 を以下の6ステップに再構成する:

1. **Step 1: テーラリング判定** -- プロジェクトのテスト資産（テストランナー、E2E フレームワーク、受入テスト計画）を検出し、実行レベルを決定
2. **Step 2: L1 実行** -- ユニットテスト + 静的解析 + ビルド + カバレッジ（現行 Step 1 と同等）
3. **Step 3: L2 実行** -- 結合テスト + E2E テスト（Entry Criteria: L1 PASS）
4. **Step 4: L3 実行** -- 受入テスト計画の検証 + US 別集計（Entry Criteria: L2 PASS または L2 スキップ時 L1 PASS）
5. **Step 5: 結果分析** -- レベル別結果の統合分析
6. **Step 6: 失敗時修正** -- 修正後は影響レベルから再実行

### 3. design.md への受入テスト計画セクション追加

design.md テンプレートに以下のセクションを追加する（「技術的アプローチ」と「リスクと注意点」の間）:

```markdown
## 受入テスト計画

### US-001: [ユーザーストーリー概要]
- **テストレベル**: L3
- **GIVEN** [前提条件] **WHEN** [検証操作] **THEN** [期待される結果]

### US-002: [ユーザーストーリー概要]
- **テストレベル**: L1/L2/L3
- **GIVEN** [前提条件] **WHEN** [検証操作] **THEN** [期待される結果]
```

### 4. spec-validator 検証項目の追加

既存の 10 項目に「11. 受入テスト計画の網羅性チェック」を追加する。spec-validator.md 内の以下の箇所を同期更新:
- 見出し: 「10 の検証項目」→「11 の検証項目」
- 検証項目リスト: 11 番目の項目を追加
- commands/spec.md の Phase 3 記述: 「10 の検証項目」→「11 の検証項目」

### 5. テンプレート同期更新（commands/spec.md と agents/spec/spec-writer.md）

design.md テンプレートと tasks.md テスト戦略テンプレートが2箇所（commands/spec.md と agents/spec/spec-writer.md）に存在するため、両方を同期更新する。

## 受入テスト計画

### US-001: /spec 実行後に受入テスト計画が design.md に含まれている
- **テストレベル**: L1
- **GIVEN** spec-writer エージェント定義に受入テスト計画生成ロジックが追加されている **WHEN** design.md テンプレートの構造を検証する **THEN** 「技術的アプローチ」と「リスクと注意点」の間に「受入テスト計画」セクションが存在する

### US-002: /test が多層構造で実行される
- **テストレベル**: L1
- **GIVEN** /test コマンド定義が多層構造に更新されている **WHEN** ワークフローの Step 構造を検証する **THEN** Step 1（テーラリング判定）、Step 2（L1）、Step 3（L2）、Step 4（L3）、Step 5（結果分析）、Step 6（失敗時修正）の6ステップが定義されている

### US-003: 各テストレベルの実行基準と合格基準が定義されている
- **テストレベル**: L1
- **GIVEN** /test コマンド定義に Entry/Exit Criteria が定義されている **WHEN** 各レベルの基準を検証する **THEN** L1, L2, L3 それぞれに Entry Criteria と Exit Criteria が明記されている

### US-004: 受入テスト結果が US 単位で集計される
- **テストレベル**: L1
- **GIVEN** /test コマンド定義に US 別集計の結果レポート形式が定義されている **WHEN** L3 結果レポートのテンプレートを検証する **THEN** US 別受入テスト結果テーブル（US, ストーリー, シナリオ数, 合格数, 結果）が定義されている

## リスクと注意点

### テンプレート不整合リスク

design.md テンプレートが commands/spec.md と agents/spec/spec-writer.md の2箇所に存在する。一方のみ更新すると不整合が発生する。tasks.md テスト戦略テンプレートも同様。

**対策**: ファイル間整合性テーブル（delta-spec.md 末尾）で同期対象を明示。横断チェックタスクで検証。

### spec-validator 項目数の不整合リスク

Phase 1 の教訓で、検証項目数の変更時に見出し・リスト・関連ファイルの記述が不整合になるリスクが判明している。

**対策**: REQ-012 で同期更新対象を明示的に列挙。タスクで全箇所を一括更新。

### 3層 vs 4層の設計判断

proposal.md は4層（L1〜L4）を要望しているが、リサーチ結果に基づき3層を採用した。ユーザーが4層を強く希望する場合、L2 を「結合テスト」、L3 を「システムテスト（E2E）」、L4 を「受入テスト」に分割する拡張が必要になる。

**対策**: テーラリングルールにより、将来の4層拡張時もテスト実行フレームワークの変更は最小限で済む設計としている。

### LLM ワークフロー特有のテスト課題

Forge のテスト対象は LLM が生成するファイル（design.md, tasks.md 等）であり、内容の完全一致テストは不安定になりやすい。構造的プロパティ（セクション存在、フォーマット準拠）のテストを推奨する。

**対策**: 受入テスト計画の検証シナリオは GIVEN/WHEN/THEN 形式で「構造的な期待結果」を記述するよう spec-writer のガイダンスに含める。
