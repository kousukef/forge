# V-Model ワークフロー強化 提案書

## 意図（Intent）

Forge の現行ワークフローはウォーターフォール的な逐次構造を持つが、実際のウォーターフォール開発（V-Model / W-Model）と比較すると、テスト戦略の多層化・要件からテストへの追跡性・各フェーズでの並行検証という3つの要素が欠落している。これらを段階的に導入し、仕様と実装の乖離を早期に検出できるワークフローに進化させる。

## スコープ（Scope）

### ユーザーストーリー

- Forge ユーザーとして、ユーザーストーリーがどの設計決定・コード・テストに対応しているか追跡したい。なぜなら仕様漏れやテスト漏れを防ぎたいから。
- Forge ユーザーとして、/spec 段階で受入テスト観点が決まっていてほしい。なぜならテストが「後付け」になると検証の質が下がるから。
- Forge ユーザーとして、各設計フェーズに対応した多層テスト（単体→結合→システム→受入）を実行したい。なぜならフラットなテストでは検出できない結合レベルの不具合があるから。
- Forge ユーザーとして、設計段階で並行的にレビュー・検証が行われてほしい。なぜなら実装後に発見される欠陥の修正コストは設計段階の10倍以上だから。

### 対象領域

- `/spec` コマンド定義（トレーサビリティ生成・受入テスト計画）
- `/test` コマンド定義（多層テスト戦略）
- `/brainstorm` コマンド定義（テスト観点の早期策定）
- `/review` コマンド定義（W-Model 左辺レビュー強化）
- `/implement` コマンド定義（トレーサビリティ更新）
- `/compound` コマンド定義（トレーサビリティアーカイブ）
- spec-writer / spec-validator エージェント
- OpenSpec スキーマ（traceability.md の追加）

## 段階的導入計画

### Phase 1: トレーサビリティ基盤

要件→設計→タスク→テストの双方向追跡を実現する。

- `/spec` で `traceability.md` を自動生成（ユーザーストーリー → 設計決定 → タスク → テスト観点のマッピング）
- spec-validator がトレーサビリティの網羅性を検証（全ストーリーにテスト観点があるか）
- `/implement` の implementer がタスク完了時にトレーサビリティを更新（実装ファイル・テストファイルのパスを記録）
- `/compound` でトレーサビリティをアーカイブ・累積スペックに反映

### Phase 2: V-Model テスト多層化

/spec 段階で受入テスト計画を策定し、/test を多層化する。

- `/spec` の design.md に「受入テスト計画」セクションを追加（ユーザーストーリーごとの検証シナリオ）
- `/test` を多層化: L1 単体テスト → L2 結合テスト → L3 システムテスト → L4 受入テスト
- 各テストレベルの定義と実行基準を明確化
- トレーサビリティと連携し、受入テスト結果をストーリー単位で集計

### Phase 3: W-Model 左辺レビュー

各設計フェーズに並行した検証活動を追加する。

- `/brainstorm` に「テスタビリティレビュー」を追加（要件がテスト可能か早期検証）
- `/spec` のフェーズゲートを強化（エントリー/エグジット基準の明確化）
- `/implement` 中の中間レビューポイント（タスク群の50%完了時点での設計適合性チェック）
- フェーズ間の品質ゲートに構造化されたチェックリストを導入

## スコープ外（Out of Scope）

- リスク管理フレームワークの導入: 有用だが独立した改善テーマ -- YAGNI
- デプロイ/リリース計画フェーズの追加: Forge は開発フェーズに集中 -- YAGNI
- ASPICE/CMMI レベルの形式的プロセス評価: 個人開発ツールには過剰 -- YAGNI
- Spiral Model 的なイテレーション構造: ウォーターフォール強化が目的 -- YAGNI

## 未解決の疑問点

- トレーサビリティの粒度: ユーザーストーリー単位か、より細かいシナリオ単位か
- 結合テスト・システムテストの具体的な実行方法（プロジェクトの技術スタックに依存する部分が大きい）
- W-Model 中間レビューのトリガー条件（タスク数ベース vs 完了率ベース）
