## 1. 分析の前提と問題設定

### 1.1 現在の議論の到達点

Discussion Report（2ラウンド8エージェント統合版）は、Forge と Obsidian Vault の統合について以下の結論に到達している:

- Vault に配置するのは **「案件横断的なナレッジ」** に絞る
- **docs/ が SSOT（正）、Vault はビュー（副）** の関係
- プロジェクト固有知識は **docs/** で完結する前提

しかし、ここに重大な理論的空白がある。**プロジェクト固有のドメイン知識――ビジネスルール、ドメインモデル、ステークホルダー要件、業界規制、技術的制約、運用知識――はどこに、どのような粒度で、どのようなライフサイクルで管理すべきか**、が未定義のまま残されている。

Discussion Report のセクション9（KM Expert による PKM 理論評価）はForge 9層構造の理論的妥当性を評価した秀逸なセクションだが、プロジェクト固有知識の「分類と配置」に関しては踏み込んでいない。本レポートはこの空白を、PKM/KM 理論の体系的適用によって埋めることを目的とする。

### 1.2 なぜこの問題が重要か

ソフトウェアエンジニアリングにおける知識の半減期に関する研究は、この問題の緊急性を示している。McDowell (1982) の古典的研究では物理学者の知識半減期を4年と推定したが、現代のソフトウェアエンジニアリングではさらに短い。Arpit Bhayani の分析によれば、ソフトウェアエンジニアの知識半減期は **2.5年から7年** の範囲にあり、World Economic Forum は技術スキルの半減期を **2.5年** と推定している。

だが、この半減期は知識の種類によって劇的に異なる。アルゴリズムの基礎やシステム設計の原理は数十年単位で有効だが、特定フレームワークの API 仕様やライブラリのゴッチャは数ヶ月で陳腐化しうる。**プロジェクト固有のドメイン知識は、この「永続知識」と「短命知識」の間のスペクトラム上に広く分布しており、一律の管理では対応できない。**

## 2. ドメイン知識の分類学（Taxonomy）

### 2.1 6類型の定義

プロジェクト固有のドメイン知識を、その性質に基づいて6つの類型に分類する。

#### 類型1: ビジネスルール（Business Rules）

- **定義:** プロジェクトが扱うドメインにおける業務上の制約、計算ロジック、条件分岐の根拠
- **例:** 「月末締め翌月15日払い」「消費税計算で1円未満は切り捨て」「承認フローは金額10万円以上で部長承認が必要」
- **特徴:** 暗黙知として関係者の頭にのみ存在することが多く、明文化の価値が極めて高い

#### 類型2: ドメインモデル（Domain Model）

- **定義:** ドメインの概念構造、エンティティ間の関係、ユビキタス言語
- **例:** 「顧客」と「アカウント」の区別、「注文」と「予約」の違い、集約ルート
- **特徴:** DDD（Domain-Driven Design）の文脈で形式化されることが多い

#### 類型3: ステークホルダー要件（Stakeholder Requirements）

- **定義:** 特定の人物・組織が持つ暗黙的・明示的な期待、優先順位、判断基準
- **例:** 「CTOはパフォーマンスを最優先する」「法務部はデータ保持期間に敏感」
- **特徴:** 属人的で、人事異動で急速に陳腐化する

#### 類型4: 業界規制（Industry Regulations）

- **定義:** 法令、業界標準、コンプライアンス要件
- **例:** GDPR、PCI DSS、医療情報の取扱い（HIPAA）、金融規制
- **特徴:** 法改正で変化するが、変化の頻度は低い。複数プロジェクトで共通することが多い

#### 類型5: 技術的制約（Technical Constraints）

- **定義:** インフラ、既存システム、技術スタック固有の制約と回避策
- **例:** 「レガシー DB が UTF-8 非対応」「API レートリミットが100req/min」「本番環境に直接 SSH 不可」
- **特徴:** プロジェクトのインフラ構成に強く依存し、インフラ刷新で無効化される

#### 類型6: 運用知識（Operational Knowledge）

- **定義:** デプロイ手順、障害対応、モニタリング、ランブック
- **例:** 「Blue-Green デプロイの切り替え手順」「OOM Killer 発動時のフォールバック」
- **特徴:** 実運用を通じてのみ獲得され、文書化されにくい

### 2.2 4軸の定量的評価

各類型を **半減期**、**移植可能性**、**抽象化可能性**、**暗黙知度** の4軸で評価する。

| 類型 | 半減期 | 移植可能性 | 抽象化可能性 | 暗黙知度 |
|---|---|---|---|---|
| ビジネスルール | 1-3年 | 低（同業界なら中） | 中（パターン化可能） | 高 |
| ドメインモデル | 2-5年 | 低-中（同ドメインなら高） | 高（DDD パターン） | 中 |
| ステークホルダー要件 | 0.5-1年 | 極低 | 低 | 極高 |
| 業界規制 | 3-10年 | 高（同業界で共通） | 高（コンプライアンスパターン） | 低 |
| 技術的制約 | 0.5-2年 | 低（同スタックなら中） | 中（制約回避パターン） | 中 |
| 運用知識 | 1-3年 | 中（類似インフラ間） | 高（ランブックパターン） | 高 |

**半減期の説明:**
- ビジネスルール（1-3年）: 業務プロセスの見直しや組織再編で変化するが、根本的なドメインロジックは比較的安定
- ステークホルダー要件（0.5-1年）: 人事異動、組織変更、戦略転換で最も速く陳腐化する
- 業界規制（3-10年）: 法改正のサイクルに依存し、ソフトウェアの知識カテゴリの中では最も安定
- 技術的制約（0.5-2年）: インフラのマイグレーションやバージョンアップで急速に無効化

**移植可能性の説明:**
- 業界規制は同業界の別プロジェクトでほぼそのまま再利用可能
- ドメインモデルは同一ドメイン（例: EC、医療、金融）であれば構造的に類似する
- ステークホルダー要件は本質的に属人的で移植不可能
- 技術的制約はスタック依存だが、同一スタックの別プロジェクトでは高い再利用価値がある

### 2.3 半減期スペクトラムの視覚化

```
短命 ← ─────────────────────────────────────────── → 長命
0.5年        1年        2年        3年        5年       10年
  │           │          │          │          │         │
  ├── ステークホルダー要件 ──┤          │          │         │
  │           ├── 技術的制約 ─┤         │          │         │
  │           │    ├── ビジネスルール ──┤          │         │
  │           │    │    ├── 運用知識 ──┤          │         │
  │           │    │    │     ├── ドメインモデル ──┤        │
  │           │    │    │     │          ├── 業界規制 ──────┤
```

この分布は、プロジェクト固有知識を一律に管理することの非合理性を示している。半減期0.5年のステークホルダー要件と、10年有効な業界規制を同じ粒度・同じ場所で管理する必然性はない。

## 3. PKM 理論の適用分析

### 3.1 Zettelkasten: 3種のノートとプロジェクトドメイン知識

Niklas Luhmann のZettelkasten（ツェッテルカステン）システムでは、ノートを **Fleeting Notes（一時的メモ）**、**Literature Notes（文献ノート）**、**Permanent Notes（永続ノート）** の3種に分類する。Sonke Ahrens の解説（「How to Take Smart Notes」）によれば、Permanent Notes は「自己完結的で、永続的に理解可能な形で書かれた」ノートであり、スリップボックスに恒久的に保管される。

さらに Ahrens は **Project Notes** を4回言及しており（Ahrens, 42, 45, 46, 71）、これらは「スリップボックスの外部に存在し、標準形式を持たない」と述べている。つまり、Zettelkasten の正統な理論では、プロジェクトノートは永続ノートとは別の存在として扱われる。

**プロジェクト固有ドメイン知識への適用:**

| ドメイン知識類型 | Zettelkasten ノート類型 | 理論的根拠 |
|---|---|---|
| ビジネスルール | **Fleeting → Literature → Permanent への昇格候補** | 初期は具体的なルールとして Fleeting に記録。パターン化されれば Permanent に昇格可能（例: 「金融ドメインにおける端数処理の一般パターン」） |
| ドメインモデル | **Literature → Permanent** | DDD の文脈で既に抽象化されていることが多く、Literature から Permanent への昇格が比較的容易 |
| ステークホルダー要件 | **Fleeting のまま留まる** | 属人的で一般化困難。プロジェクト終了とともに意味を失うため、Permanent に昇格する価値が低い |
| 業界規制 | **Permanent（初期から）** | 案件横断で適用可能なため、最初から Permanent Note として作成する価値がある |
| 技術的制約 | **Fleeting → Literature** | 具体的な制約自体は短命だが、「制約への対処パターン」は Permanent に昇格可能 |
| 運用知識 | **Literature → Permanent への昇格候補** | ランブックの具体的手順は Literature だが、「障害対応の一般的アプローチ」に抽象化されれば Permanent |

**重要な理論的洞察:** Zettelkasten において、プロジェクト固有ドメイン知識の大部分は **Permanent Notes に直接なるべきものではない**。それらは Fleeting または Literature の段階で「プロジェクトスコープ」に留まるべきであり、Permanent に昇格するのは **意図的な抽象化作業** を経た後のみである。

この洞察は Discussion Report の「docs/ が SSOT、Vault はビュー」という設計と理論的に整合する。docs/ に蓄積されるプロジェクト固有知識は Zettelkasten における Fleeting/Literature Notes に相当し、Vault の Permanent Notes とは本質的に異なる存在である。

### 3.2 PARA: プロジェクト完了後のライフサイクル

Tiago Forte の PARA メソッドでは情報を **Projects（プロジェクト）**、**Areas（責任領域）**、**Resources（リソース）**、**Archives（アーカイブ）** の4カテゴリに分類する。

PARA の核心的原則は「情報を生成のコンテキストではなく、**使用のコンテキスト**で整理する」ことにある。Forte の言葉を借りれば「プロジェクトが完了したら、その内容は Areas、Resources、または Archives に移動する」。

**プロジェクト固有ドメイン知識の PARA ライフサイクル:**

```
Phase 1: アクティブプロジェクト期
  全ドメイン知識 → Projects/ に配置
  （Forge の文脈: docs/ + openspec/）

Phase 2: プロジェクト完了時（PARA の「移動」が発生）
  ├── ビジネスルール（一般化済み） → Resources/
  ├── ドメインモデル → Resources/ または Areas/（継続保守の場合）
  ├── ステークホルダー要件 → Archives/（再利用価値なし）
  ├── 業界規制 → Areas/（継続的な責任領域として）
  ├── 技術的制約 → Archives/（スタック変更で無効化）
  └── 運用知識 → Areas/（保守継続）or Archives/（完全移管）

Phase 3: 長期保管
  Resources/ のうち参照されなくなったもの → Archives/
  Areas/ のうち責任が離れたもの → Archives/
```

**Forge への適用上の洞察:** 現在の Forge は OpenSpec の `archive/` で変更単位のアーカイブを行っているが、**プロジェクト全体の完了時に docs/ 内のドメイン知識をどう処理するか**のライフサイクルが定義されていない。PARA 理論に基づけば、プロジェクト完了時に各ドメイン知識を「Resources（案件横断で再利用可能）」「Areas（継続責任）」「Archives（凍結保管）」に分類し移動する作業が必要となる。

Discussion Report が提案する Vault の `10_Projects/` はまさに PARA の Projects/Areas に対応し、`20_Tech/` は Resources に対応する。しかし、**Projects → Resources/Archives への移動タイミングと方法** が未設計である。

### 3.3 LYT: MOC によるプロジェクト固有知識のナビゲーション

Nick Milo の LYT（Linking Your Thinking）フレームワークの中核は MOC（Map of Content）である。MOC は「ノートのリストではなく関係性の地図」であり、「Mental Squeeze Point（認知的圧縮点）」――未整理の知識が混沌として圧倒される状態――を解消するためのツールである。

LYT の重要な原則の一つは、**MOC は事前に作成するものではなく、ノートが一定数蓄積してから自然発生的に生まれるべき** というものである。Discussion Report のセクション9.2 でも「空の MOC を先に作らない。ノートが10件蓄積してから」と指摘されており、これは LYT の原則に忠実である。

**プロジェクト固有知識への LYT 適用:**

プロジェクト固有のドメイン知識に対して MOC が最も有効に機能するのは、**プロジェクトの中期以降**、すなわち知識が十分に蓄積され関係性が見え始める段階である。

```
プロジェクト開始期:
  個別ノートを docs/ に蓄積（MOC 不要）

プロジェクト中期（ノート10件以上）:
  MOC 候補が自然発生
  例: docs/domain-map.md として
    - ビジネスルール間の関係
    - エンティティ間の依存関係
    - 技術的制約とビジネスルールの相互作用

プロジェクト完了・知識の卒業時:
  プロジェクト MOC の一部が Vault の Domain MOC に昇格
  例: ~/vault/00_MOC/ECDomain.md
    - EC ドメインの一般的パターンへのリンク集
    - プロジェクト A, B, C で得たドメイン知見の統合
```

**3層 MOC 構造の提案:**

```
Level 1: Home.md（全体の入口）
  │
Level 2: Domain MOC（ドメイン別）
  │  例: ECDomain.md, HealthcareDomain.md, FinanceDomain.md
  │
Level 3: Topic MOC（トピック別）
     例: AuthenticationPatterns.md, PaymentProcessing.md
```

プロジェクト固有の docs/ 内ドメイン知識は、Level 3 の Topic MOC から参照される構造とする。これにより、特定プロジェクトの文脈に閉じた知識と、案件横断の一般化された知識の間にナビゲーションパスが生まれる。

### 3.4 CODE: Capture から Express までの変容過程

Tiago Forte の CODE フレームワーク（Capture → Organize → Distill → Express）は、知識が生成されてから活用されるまでの4段階の変容を記述する。

**プロジェクト固有ドメイン知識の CODE 適用:**

| CODE 段階 | プロジェクトドメイン知識の文脈 | Forge/Vault での対応 |
|---|---|---|
| **Capture** | 開発中に発見したドメインルール、制約、ゴッチャを即座に記録 | `/compound` による docs/compound/ への自動蓄積。**Progressive Summarization の Layer 1（原文記録）** |
| **Organize** | 知識をプロジェクト内の適切な場所に配置し、案件横断性を判定 | docs/ 内の分類 + Vault 40_Inbox/ への投入。**PARA の Projects/Resources/Areas 振り分け** |
| **Distill** | 具体的な知見を抽象化し、パターン化する | Vault でのノート整理。**Progressive Summarization の Layer 2-4（ボールド→ハイライト→サマリー）** |
| **Express** | 蒸留された知識を次のプロジェクトで活用する | `/brainstorm` での Vault 参照。**知識の「表現」としての proposal.md 品質向上** |

**Progressive Summarization のプロジェクトドメイン知識への適用:**

Forte の Progressive Summarization は5層の段階的蒸留を提唱する:

1. **Layer 0（原本）:** /compound が docs/compound/ に書き出す生の教訓
2. **Layer 1（記録済み）:** docs/ 内のドメイン知識ファイル（domain-rules.md 等）
3. **Layer 2（ボールド化）:** Vault に移動した知識の重要部分を太字化
4. **Layer 3（ハイライト化）:** 太字部分のうち核心的主張をハイライト
5. **Layer 4（エグゼクティブサマリー）:** 1-2文でのエッセンス抽出

**重要な洞察:** Forte は「すべてのノートが Layer 4 に到達する必要はない」と強調している。実際、Progressive Summarization は **Just-In-Time** で行われるべきであり、「将来使うかもしれない」という投機的動機で全ノートを蒸留するのは時間の無駄である。この原則は、Discussion Report の「Vault の整理は週1回10分」「40_Inbox/ の放置を許容する設計」と完全に一致する。

## 4. 「知識の卒業」概念

### 4.1 卒業の定義

本レポートでは「知識の卒業（Knowledge Graduation）」を以下のように定義する:

> **知識の卒業とは、プロジェクトスコープに閉じていた知識が、意図的な抽象化・一般化を経て、案件横断の個人ナレッジベースに移行するプロセスである。**

これは Zettelkasten における Fleeting/Literature → Permanent への昇格、PARA における Projects → Resources/Areas への移動、LYT におけるプロジェクト固有ノートから Domain MOC へのリンク化に相当する。

### 4.2 卒業の3条件

ドメイン知識が「卒業」するためには、以下の3つの変容を経る必要がある:

#### 条件1: 具体 → 抽象（Concrete → Abstract）

- **変容前:** 「案件Aでは消費税計算時に1円未満を切り捨てる」
- **変容後:** 「金融ドメインの端数処理パターン: 切り捨て/切り上げ/四捨五入の選択基準とその法的根拠」
- **検証:** 元のプロジェクト名を削除しても意味が通じるか

#### 条件2: 固有 → 一般化（Specific → Generalized）

- **変容前:** 「NextAuth Prisma Adapter v5 では createdAt が nullable に変更」
- **変容後:** 「ORM アダプターのメジャーバージョンアップ時にはスキーマ変更の確認が必須。expand-contract パターンで段階移行」
- **検証:** 異なる技術スタックでも適用可能な知見が含まれるか

#### 条件3: コンテキスト付き → コンテキストフリー（Context-Bound → Context-Free）

- **変容前:** 「CTO が承認フローを3階層に指定した（2026年1月MTGにて）」
- **変容後:** 「承認フロー設計時の考慮点: ステークホルダーの組織構造がフロー階層を規定する」
- **検証:** 元のステークホルダーを知らない人が読んでも有用か

### 4.3 卒業タイミングの判定

卒業は以下の **トリガーイベント** によって発生する:

| トリガー | 説明 | 対応する Forge アクション |
|---|---|---|
| **プロジェクト完了** | プロジェクトクローズ時の知識棚卸し | `/compound` の最終実行（プロジェクト完了版） |
| **2回目の遭遇** | 同じパターンの知識が別プロジェクトで出現 | `/brainstorm` 時に「以前も似た状況があった」と認識 |
| **反復的参照** | 同じ docs/ ノートが3回以上参照される | Vault の `last_referenced` カウンターで検知 |
| **技術ブログ化** | 知識を外部に発信する際に一般化が自然に発生 | Express 段階（CODE の最終ステップ） |

**最も確実な卒業トリガーは「2回目の遭遇」** である。Zettelkasten の創始者 Luhmann は「驚きの原理（Prinzip der Uberraschung）」を重視した。つまり、スリップボックスにノートを追加する際に、既存のノートとの予期しない接続が発見される瞬間こそ、知識が新たな段階に昇格するタイミングである。

### 4.4 卒業の自動化可能性

Discussion Report のセクション4.1 では `/compound` 実行時の案件横断性ヒューリスティック判定が提案されている:

```
技術名を含む → 高（卒業候補）
「～すべきでない」形式 → 高（卒業候補）
ビジネスロジック固有 → 低（卒業しない）
プロジェクト固有設定値 → 低（卒業しない）
```

このヒューリスティックは妥当な出発点だが、PKM 理論の観点からは以下の改善が必要である:

1. **「2回目の遭遇」検知の自動化:** `/brainstorm` で Vault を参照した際に、docs/ 内の知識と Vault 内の知識の類似性を検出する仕組み。もし類似の知見が既に docs/ にあり、かつ Vault にはないなら、それは「まだ卒業していない知識が2度目に遭遇された」状態であり、卒業の強いシグナルとなる。

2. **参照カウントによる自然な卒業:** docs/ 内の特定ノートが `/brainstorm` や `/spec` で3回以上参照された場合、そのノートを Vault 40_Inbox/ への卒業候補としてフラグする。これは LYT の「Mental Squeeze Point」の自動検知に相当する。

3. **卒業の変容支援:** 卒業時に AI が「具体→抽象」「固有→一般化」「コンテキスト付き→コンテキストフリー」の3変容を支援するプロンプトテンプレートを提供する。ただし、**最終的な変容の判断と実行は人間が行う**（「人間が書き、AI が読む」原則の遵守）。

### 4.5 「卒業しない知識」の重要性

PKM 理論においてしばしば見落とされるのは、**すべての知識が永続ノートに昇格すべきではない** という事実である。Zettelkasten においても、Luhmann のスリップボックスに入らなかったノートは膨大にあった。

プロジェクト固有のステークホルダー要件や、特定バージョンの技術的制約は、プロジェクトライフタイムを超えて保持する価値がない場合が多い。これらは **「卒業せずに Archives で安らかに眠る」** ことが正しい帰結である。

Discussion Report の設計では、docs/compound/ に全ての学びを蓄積し、その中から案件横断性の高いものだけを Vault に投入する。これは「卒業しない知識」を無理に Vault に持ち込まない設計であり、理論的に正しい。

## 5. Single Vault vs Multi-Vault 戦略

### 5.1 3つの戦略の理論的分析

#### 戦略A: Single Vault（全知識を1つの Vault に集約）

**理論的背景:** Zettelkasten の原理に最も忠実。Luhmann は1つのスリップボックスに全ノートを入れ、番号体系とリンクでナビゲートした。LYT もまた Single Vault を前提として設計されている。

**長所:**
- 知識間の予期しない接続（セレンディピティ）が最大化される
- 双方向リンクとグラフナビゲーションの価値が最大になる
- 管理コストが低い（設定、プラグイン、テーマが1セット）
- 検索が一元化される

**短所:**
- Vault が大規模化するとパフォーマンスが劣化する（Obsidian フォーラムの大規模 Vault ユーザーの報告）
- プロジェクト間のコンテキスト切替がノイジーになる
- 機密性の異なるプロジェクト情報が混在するリスク

**Obsidian コミュニティの実践知:** フォーラムでの議論によると、Single Vault のユーザーは「ノートが増えるにつれて Vault 自体が引力（gravity）を持ち、分割の摩擦を超える」と報告している。一方で「プロジェクトノートとエバーグリーンノートの増加速度が10倍違い、混在が苦痛になる」という証言もある。

#### 戦略B: Multi-Vault（プロジェクトごとに独立 Vault）

**理論的背景:** PARA の Projects カテゴリに対応。各プロジェクトが独立した知識空間を持つ。

**長所:**
- プロジェクト間の関心の分離が明確
- 各 Vault の設定（プラグイン、テーマ、タグ戦略）を最適化できる
- 機密情報の隔離が容易
- Vault サイズが小さく保たれる

**短所:**
- **Vault 間のリンクが不可能**（Obsidian の根本的制約）
- 案件横断の知識発見が著しく困難
- デイリーノートの分散
- 設定の重複管理

**Obsidian コミュニティの実践知:** Multi-Vault ユーザーは「プロジェクト Vault とメイン Vault の間で手動コピーが必要で、実質的に卒業プロセスが強制される」と報告している。これは「知識の卒業」の強制機構として機能しうるが、摩擦が高すぎて放棄されるリスクもある。

#### 戦略C: ハイブリッド（Forge docs/ + 中央 Vault）

**理論的背景:** Discussion Report が提案する戦略。プロジェクト固有知識は Forge の docs/（ファイルシステム上のプロジェクトディレクトリ）に、案件横断知識は中央 Vault に配置する。

**長所:**
- プロジェクト固有知識と案件横断知識の自然な分離
- docs/ は Git 管理下で、コードと同期してバージョン管理される
- 中央 Vault の知識グラフは案件横断ノートのみで構成され、ノイズが少ない
- Forge の既存ワークフロー（/compound → docs/）との親和性が高い

**短所:**
- docs/ と Vault 間の知識移動（卒業）が手動
- docs/ 内の知識は Obsidian のグラフ・バックリンク機能の恩恵を受けない
- 二重管理のリスク（Discussion Report が指摘する SSOT 問題）

### 5.2 理論的推奨

PKM 理論を総合すると、**戦略C（ハイブリッド）が最も理論的に妥当** である。その根拠を以下に述べる。

1. **Zettelkasten の観点:** Permanent Notes のみがスリップボックスに入り、Project Notes は外部に存在する。Forge の docs/ が Project Notes、Vault が Zettelkasten（Permanent Notes の集合）に対応する。この分離は Luhmann の原設計に忠実。

2. **PARA の観点:** Projects（docs/）と Resources/Areas（Vault）の分離は PARA の4カテゴリ分離と整合。プロジェクト完了時に docs/ の知識を Vault に「卒業」させるプロセスは PARA の「Archives への移動」に相当。

3. **LYT の観点:** MOC は案件横断の知識に対して最も価値を発揮する。プロジェクト固有の MOC は docs/ 内で十分機能し、Vault の MOC は案件横断のドメインパターンへのナビゲーションに特化すべき。

4. **CODE の観点:** Capture と Organize は docs/ で発生し（/compound の自動蓄積）、Distill と Express は Vault で発生する（人間の整理 → /brainstorm での参照）。この分離は CODE の4段階の自然な境界に一致する。

5. **実用的配慮:** Forge の docs/ は Git 管理下にあり、コードのバージョン管理と同期する。これはソフトウェアプロジェクトにおけるドメイン知識管理の基本要件（コードと仕様の同期）を満たす。Vault にこの役割を移すと、Git との二重管理が発生する。

### 5.3 戦略C の強化提案: Nested Vault のオプション

Obsidian フォーラムの議論で報告されている **Nested Vault** 戦略を応用すれば、docs/ を Vault のサブディレクトリとして配置することも技術的には可能である。これにより、docs/ 内のノートも Obsidian のグラフ・バックリンクの恩恵を受けられる。

```
~/vault/
├── 00_MOC/
├── 10_Projects/
│   ├── forge/  → symlink to ~/forge/docs/  (または soft link)
│   └── other-project/  → symlink
├── 20_Tech/
└── 40_Inbox/
```

ただし、この構成はシンボリックリンクの管理や Git の .gitignore 設計を複雑化させるため、Phase 3 以降に検討する「上級者向けオプション」として位置づけるべきである。Discussion Report の段階的導入計画（Phase 0 → 1 → 2 → 3）の精神に沿えば、初期段階ではシンプルなハイブリッド戦略を採用し、運用実績を蓄積してから Nested Vault を検討すべきである。

## 6. AI + PKM の新パラダイム

### 6.1 従来の PKM の前提

Zettelkasten、PARA、LYT、CODE のすべてのフレームワークは、**「人間が書き、人間が読む」** という暗黙の前提の上に構築されている。

- Luhmann は手書きでカードを作成し、自分で読み返した
- Forte は自分のノートを自分でブラウズし、Progressive Summarization を自分で行った
- Milo は自分の MOC を自分でナビゲートし、リンクを自分で作成した

この前提において、ノートの品質基準は「未来の自分が読んで理解できるか」であり、ノートの発見可能性は「人間の記憶とナビゲーション能力」に依存していた。

### 6.2 Forge が導入する新しい前提

Forge は2つの新しい知識フローを導入する:

#### フロー1:「人間が書き、AI が読む」

- **人間** が Obsidian Vault に知識を整理・記述する
- **AI**（/brainstorm の spec-writer Agent）が MCP 経由でその知識を読み取り、proposal.md に反映する

このフローにおけるノート品質基準は従来と異なる:

| 品質基準 | 人間が読む場合 | AI が読む場合 |
|---|---|---|
| 文体 | 自然で読みやすい散文 | 構造化された Markdown（AI パース容易） |
| コンテキスト | 暗黙の前提を共有 | **明示的なメタデータ**（フロントマター）が必須 |
| リンク | 連想的で主観的 | **セマンティックに明確なリンク**が望ましい |
| 粒度 | 人間の注意力に合わせた長さ | **アトミック**（200行以下）がトークン効率に直結 |
| 発見可能性 | グラフのビジュアル探索 | **MOC + タグ + セマンティック検索** |

**重要な帰結:** 「AI が読む」前提では、ノートのフロントマター設計が従来の PKM より重要になる。Discussion Report で提案されている `type`, `domain`, `tags`, `source_file`, `status`, `last_validated` のフロントマターは、AI ナビゲーションのために不可欠である。

#### フロー2:「AI が書き、人間が整理する」

- **AI**（/compound Agent）が学びを docs/compound/ に書き出し、案件横断ノートのドラフトを Vault 40_Inbox/ に生成する
- **人間** が週1回のレビューで整理・リンク付与・status 更新を行う

このフローは従来の PKM にはなかった「**AI による Capture の自動化**」である。CODE フレームワークの Capture 段階を AI に委譲し、人間は Organize 以降に集中できる。

**PKM 理論的評価:** 従来の PKM で最も挫折しやすいのは Capture の習慣化である（Pragmatist の Web 調査: 「最初の30日で毎日キャプチャする習慣が定着しなければシステムは放棄される」）。AI による Capture の自動化は、この最大のボトルネックを解消する。

ただし、**AI が生成したドラフトを人間が「自分の言葉で」書き直す** ステップを省略してはならない。Zettelkasten の根本原則は「自分の言葉で書くことで理解が深まる」であり、AI 生成のドラフトをそのまま Vault に放置することは、PKM の本質的価値を毀損する。Discussion Report の `status: draft → validated` のステータス遷移は、この原則を制度化するものとして理論的に正しい。

### 6.3 AI + PKM が変えるプロジェクト固有知識の最適配置

従来の PKM では、プロジェクト固有知識の大部分は「整理のコストに見合わない」として放置されがちだった。しかし AI によるCapture 自動化 + セマンティック検索の組み合わせは、この計算を根本的に変える:

1. **Capture コストの劇的低下:** /compound による自動蓄積で、人間が手動で書き留める必要がない
2. **検索コストの劇的低下:** セマンティック検索 + グラフ走査で、人間が「あのノートどこだっけ」と探す必要がない
3. **Distill のジャストインタイム化:** AI が /brainstorm 時にノートを読み取る際、必要な部分だけを自動的に抽出する

**結論:** AI + PKM パラダイムでは、従来「整理する価値がない」と判断されていたプロジェクト固有知識の一部が、**低コストで蓄積・低コストで検索・低コストで活用** できるようになる。これは Discussion Report が提案する「docs/ が SSOT」の戦略をさらに強化する根拠となる。docs/ に「やや雑でも良いから」ドメイン知識を蓄積しておけば、AI が必要な時に必要な部分だけを取り出してくれるからである。

### 6.4 セレンディピティの機械化

Discussion Report のセクション9.3 で指摘された「Emergence（創発）の設計不足」は、AI + PKM の文脈で新しい解決策を得る。

従来の PKM では、セレンディピティ（偶然の発見）はユーザーがグラフビューを眺めたり、関連ノートを偶然クリックしたりすることで生じていた。Forge では、これを `/brainstorm` 時のランダム弱関連ノート提示（「セレンディピティモード」）として機械化できる。

```
/brainstorm 実行時:
  1. 関連度の高い Vault ノートを取得（通常の参照）
  2. 関連度の低い Vault ノートをランダムに1-2件取得（セレンディピティ）
  3. セレンディピティノートが提案に影響した場合、
     proposal.md に「セレンディピティ発見」セクションとして記録
```

この設計は Luhmann が Zettelkasten に求めた「コミュニケーションパートナーとしてのスリップボックス」を、AI エージェントによって実現するものである。Luhmann は「スリップボックスは著者が思い出せないことを思い出させてくれる」と述べたが、AI Agent + Vault はまさにこの機能を提供する。

## 7. 理論に基づく知識配置フレームワークの提案

### 7.1 ドメイン知識の分類と配置先の対応表

セクション2の分類学とセクション3-6の理論分析を統合し、以下の配置フレームワークを提案する。

| ドメイン知識類型 | 生成時配置 | 卒業後配置 | 卒業条件 | 配置の理論的根拠 |
|---|---|---|---|---|
| ビジネスルール | `docs/domain-rules.md` | `~/vault/20_Tech/patterns/` | 同パターンが2回出現 | Zettelkasten: Fleeting→Permanent（抽象化を経て） |
| ドメインモデル | `docs/domain-model.md` | `~/vault/10_Projects/<domain>/` | プロジェクト完了時の棚卸し | PARA: Projects→Resources |
| ステークホルダー要件 | `docs/stakeholders.md` | **卒業しない**（Archives） | -- | Zettelkasten: Fleeting のまま廃棄が正しい |
| 業界規制 | `~/vault/20_Tech/regulations/` **（直接 Vault）** | -- | 初期から案件横断 | PARA: Areas（継続的責任） |
| 技術的制約 | `docs/tech-constraints.md` | `~/vault/20_Tech/gotchas/` | 制約パターンとして一般化 | CODE: Distill（具体→抽象） |
| 運用知識 | `docs/runbooks/` | `~/vault/20_Tech/patterns/` | ランブックパターンとして一般化 | LYT: Topic MOC への統合 |

**注目点:** 業界規制は初めから Vault に直接配置する。これは PARA における Areas（継続的な責任領域）に該当し、プロジェクトスコープに閉じる理由がないためである。Discussion Report では docs/ を SSOT としているが、業界規制についてはこの原則の例外として Vault を SSOT とすべきである。

### 7.2 知識のライフサイクルと配置場所

```
生成（Capture）            蓄積（Organize）           活用（Distill/Express）
─────────────────────────────────────────────────────────────────────────

/compound 自動蓄積  ──→  docs/compound/learnings/  ──→  /brainstorm で参照
                         （SSOT: 全ての学び）            （docs/ 内の知識）
                              │
                              │ 案件横断性判定
                              ↓
                         Vault 40_Inbox/             ──→  /brainstorm で参照
                         （AI ドラフト）                    （Vault 内の知識）
                              │
                              │ 人間が週1回整理
                              ↓
                         Vault 20_Tech/              ──→  次のサイクルへ
                         （Permanent Note）
                              │
                              │ 3回以上参照
                              ↓
                         Skills/ 化検討               ──→  Forge Skills に昇格
                         （最も成熟した知識）
```

**知識の成熟モデル:**

```
Level 0: 暗黙知（頭の中にのみ存在）
  ↓  /compound で言語化
Level 1: docs/compound/（生の記録、プロジェクト固有）
  ↓  案件横断性判定 + 人間の整理
Level 2: Vault 20_Tech/（一般化済み、案件横断）
  ↓  3回以上参照 + パターン化
Level 3: Skills/（実行可能なコード化）
```

この4段階は、Zettelkasten の Fleeting → Literature → Permanent に1段階追加したものである。Skills への昇格は Forge 固有の概念であり、「知識のコード化」という従来の PKM にはない最終段階を表す。Discussion Report の付録D で言及されている「Vault → Skill 昇格パイプライン（3回以上参照されたらフラグ）」は、この成熟モデルの最終段階に対応する。

### 7.3 Progressive Summarization のプロジェクトドメイン知識への具体的適用

Forte の Progressive Summarization の5層をプロジェクトドメイン知識に適用する:

**例: 「NextAuth + Prisma Adapter の認証実装」に関するドメイン知識**

```
Layer 0（原体験）:
  開発中に Prisma Adapter v5 で createdAt が nullable に変更されて
  マイグレーション失敗を経験。3時間のデバッグで解決。

Layer 1（/compound による記録）: docs/compound/learnings/2026-03-02.md
  NextAuth Prisma Adapter v5 にアップグレードすると、createdAt カラムが
  NOT NULL → nullable に変更される。既存データがある場合、マイグレーション失敗。
  対処法: expand-contract パターンで段階的に移行する。

Layer 2（Vault 40_Inbox/ → 20_Tech/gotchas/ に移動、ボールド化）:
  NextAuth Prisma Adapter v5 にアップグレードすると、**createdAt カラムが
  NOT NULL → nullable に変更される**。既存データがある場合、**マイグレーション失敗**。
  対処法: **expand-contract パターン**で段階的に移行する。

Layer 3（ハイライト化、一般化の萌芽）:
  ==ORM アダプターのメジャーバージョンアップ時にはスキーマ変更が発生しうる。==
  **createdAt が NOT NULL → nullable** に変更。
  ==対処法: expand-contract パターン==で段階的に移行。

Layer 4（エグゼクティブサマリー、完全に一般化）:
  > ORM アダプターのバージョンアップ時は、スキーマの互換性を必ず事前確認する。
  > 既存データがある場合は expand-contract パターンで段階移行が安全。
```

Layer 1-2 はプロジェクト固有の文脈を保持しており docs/ および Vault 20_Tech/ に適する。Layer 3-4 は一般化されており、将来的に Skills 化の候補となる。

**重要:** Progressive Summarization は **必要になった時にだけ次の層に進む** べきである。Layer 1 の記録を全て Layer 4 まで蒸留する必要はない。次のプロジェクトで「Prisma Adapter」に関する /brainstorm を実行した時点で、AI が Layer 2 のノートを読み取り、必要な情報を抽出する。人間が Layer 4 まで蒸留するのは、そのノートが何度も参照され、Skill 化を検討する段階に至った場合のみで十分である。

### 7.4 フレームワーク適用のガイドライン

以下の判定フローを提案する:

```
ドメイン知識が発生
  │
  ├─ 案件横断か?
  │   ├─ Yes（業界規制等）→ Vault に直接配置
  │   └─ No → docs/ に配置
  │
  │  [プロジェクト進行中]
  │
  ├─ /compound で記録されるか?
  │   ├─ Yes → docs/compound/ に自動蓄積（SSOT）
  │   │   └─ 案件横断性ヒューリスティック判定
  │   │       ├─ 高 → Vault 40_Inbox/ にドラフト
  │   │       ├─ 中 → ユーザーに確認
  │   │       └─ 低 → docs/ のみ
  │   └─ No → docs/ の適切なファイルに手動記録
  │
  │  [プロジェクト完了時]
  │
  ├─ 知識の棚卸し実施
  │   ├─ ビジネスルール → パターン化できるか検討 → Yes: Vault に卒業
  │   ├─ ドメインモデル → 同ドメインの次案件に適用可能か → Yes: Vault に卒業
  │   ├─ ステークホルダー要件 → Archives（卒業しない）
  │   ├─ 技術的制約 → gotcha パターンとして一般化 → Yes: Vault に卒業
  │   └─ 運用知識 → ランブックパターンとして一般化 → Yes: Vault に卒業
  │
  │  [長期運用]
  │
  └─ Vault ノートが3回以上参照 → Skills 化検討
```

## 8. Discussion Report への理論的フィードバック

### 8.1 高く評価する点

1. **「docs/ が SSOT、Vault はビュー」の設計判断:** Zettelkasten における Project Notes と Permanent Notes の分離に理論的に整合している。これは PKM 理論の正しい適用である。

2. **「人間が書き、AI が読む」原則:** Discussion Report の KM Expert が指摘した通り、この原則は Zettelkasten, PARA, GTD, LYT の全フレームワークと整合する。加えて、AI による Capture 自動化（/compound → 40_Inbox/）は CODE フレームワークの最大のボトルネックを解消する革新的設計である。

3. **段階的導入（Phase 0 → 1 → 2 → 3）:** Progressive Summarization の「投機的整理をしない」原則と整合する。必要性が実証されてから次の段階に進む設計は、PKM 放棄リスクの最大の緩和策である。

4. **陳腐化管理の knowledge_type 別閾値:** セクション2で分析した「半減期のスペクトラム」と一致しており、gotcha 3ヶ月、ADR 6ヶ月、pattern 6-12ヶ月の設定は理論的に妥当である。

5. **30_Daily/ の不採用:** デイリーノートの PKM 放棄リスクに関する判断は、コミュニティの実践知と一致しており正しい。

### 8.2 改善を推奨する点

1. **プロジェクト完了時の知識棚卸しプロセスが未設計:**
   PARA 理論では Projects → Resources/Areas/Archives への移動がプロジェクトライフサイクルの重要なステップである。Discussion Report には、プロジェクト完了時に docs/ 内のドメイン知識をどう棚卸しするかの設計がない。**プロジェクトクローズ時の `/compound` に「知識棚卸しモード」を追加** し、docs/ 内のドメイン知識ファイルを走査して卒業候補を提案する機能を推奨する。

2. **業界規制の配置戦略が不明確:**
   業界規制はプロジェクト固有ではなく、初期から案件横断である。Discussion Report では全てのドメイン知識を「まず docs/ に」と想定しているが、業界規制については **Vault を SSOT** とし、docs/ からは Vault のノートを参照する構造（ポインタ配置）が理論的に正しい。

3. **10_Projects/ の役割が不明確:**
   Vault の `10_Projects/` が「案件別ドメイン知識」とされているが、これは docs/ との二重管理リスクを内包する。理論的整合性のためには、10_Projects/ は **プロジェクト完了後に卒業した知識の「アーカイブ兼インデックス」** として位置づけ、アクティブプロジェクトの知識は docs/ に一元化すべきである。具体的には:
   - アクティブプロジェクト: docs/ のみ（10_Projects/ にはプロジェクトの index.md のみ配置し、docs/ へのポインタとする）
   - 完了プロジェクト: 棚卸し後に卒業した知識を 10_Projects/ に移動

4. **「知識の卒業」トリガーの自動化が未設計:**
   セクション4.4 で提案した「2回目の遭遇検知」「参照カウントによる卒業シグナル」の自動化は、/brainstorm 統合（Phase 2）の時点で実装可能であり、Phase 2 のタスクリストに追加を推奨する。

5. **セレンディピティモードの優先度を上げるべき:**
   Discussion Report の付録D に「今後の検討事項」として記載されているが、LYT と Zettelkasten の理論的観点から、セレンディピティ（予期しない接続の発見）は PKM の本質的価値である。Phase 2 の /brainstorm 統合に「scoreThreshold を低く設定した弱関連ノート1件の提示」を含めることを推奨する。実装コストは低い（graph(search-traverse) の scoreThreshold パラメータを調整するのみ）。

6. **Progressive Summarization のワークフローへの明示的組み込み:**
   週1回10分の Vault 整理時に何をすべきかが曖昧である。Progressive Summarization の Layer 2（ボールド化）と Layer 3（ハイライト化）を整理作業の具体的ステップとして定義することで、「何をすればよいかわからない → 放棄」のリスクを低減できる:
   - **週次整理の具体的手順:**
     1. 40_Inbox/ のドラフトを確認（2分）
     2. 有用なノートを 20_Tech/ の適切なフォルダに移動（3分）
     3. 移動したノートの重要部分をボールド化（Layer 2）（3分）
     4. status: draft → validated に変更（1分）
     5. 明らかに不要なノートを削除（1分）

### 8.3 理論的に見落とされている論点

1. **「知識の死」の設計:**
   陳腐化管理は `last_validated` で対応しているが、**明示的に「この知識は死んだ（obsolete）」とマークし、参照対象から除外する** メカニズムが未設計。dead ステータスのノートは /brainstorm の参照から除外し、年次で物理削除する設計を推奨する。

2. **知識の粒度と Forge の層の対応:**
   現在の Discussion Report は「Vault ノートはアトミック（200行以下）」としているが、docs/ 内のドメイン知識ファイルの粒度基準がない。docs/ に関しても「1ファイル = 1つのドメイン概念」のアトミック原則を適用し、ファイルが肥大化した場合は分割する基準を設けるべきである。

3. **タグ戦略の PKM 理論整合性:**
   Discussion Report ではフロントマターに `domain`, `tags` が定義されているが、タグの設計方針が未記述である。Zettelkasten ではタグは「分類」ではなく「再発見のためのフック」として使うべきとされる。LYT ではタグよりもリンク（[[]]）を重視する。タグとリンクの使い分け方針を明確にすべきである。

## 総括

本分析の核心的結論は以下の3点に集約される:

**第一に、プロジェクト固有ドメイン知識は一枚岩ではない。** 6つの類型はそれぞれ異なる半減期・移植可能性・抽象化可能性を持ち、一律の管理では対応できない。業界規制は初期から Vault に配置すべきであり、ステークホルダー要件は卒業させるべきではない。

**第二に、Discussion Report の「docs/ が SSOT、Vault はビュー」という設計は PKM 理論的に正しい。** これは Zettelkasten における Project Notes と Permanent Notes の分離に対応し、PARA の Projects と Resources の区別に対応し、CODE の Capture/Organize と Distill/Express の境界に対応する。ただし、プロジェクト完了時の知識棚卸しプロセスが未設計であり、この追加が不可欠である。

**第三に、AI + PKM は従来の PKM フレームワークの前提を変える。** AI による Capture の自動化は CODE の最大のボトルネックを解消し、セマンティック検索はノートの発見可能性を劇的に向上させ、セレンディピティモードは Zettelkasten の「驚きの原理」を機械化する。この新パラダイムにおいて、プロジェクト固有ドメイン知識は「整理のコストが高いので放置」から「低コストで蓄積し AI に活用させる」へと戦略が転換される。docs/ にやや雑にでもドメイン知識を蓄積しておけば、AI が必要な時に必要な部分だけを取り出す――これが AI 時代のプロジェクトドメイン知識管理の核心である。
